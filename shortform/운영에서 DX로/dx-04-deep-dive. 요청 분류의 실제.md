# Deep Dive: 요청 분류의 실제

## 네 가지 분류는 어떤 판단으로 작동했는가

> 본편: [04. 일 분석 프로세스](https://sprawlingdev.substack.com/p/04)

**04편에서 요청을 네 가지로 분류하는 구조를 소개했습니다. 이 글에서는 각 분류가 실제로 어떤 판단 기준과 전략으로 작동했는지를 정리합니다.**

---

### 네 가지 분류를 다시 봅니다

요청이 들어오면 저희는 다음 순서로 해결 방식을 판단했습니다.

`요청` → `분석` → `팀 의사결정` → `해결 방식 선택`

해결 방식은 네 가지로 고정했습니다.

- **프로세스 변화로 해결할 수 있는가** — 요청부서의 업무 절차, 승인 경로, 역할 분담을 재정의해서 해결하는 방식입니다.
- **자동화로 해결할 수 있는가** — 규칙 기반 반복 업무를 RPA, n8n 등으로 자동화하는 방식입니다.
- **기존 플랫폼이나 SaaS로 해결할 수 있는가** — 이미 도입된 시스템의 미활용 기능이나 외부 솔루션으로 해결하는 방식입니다.
- **직접 개발** — 위 세 가지로 해결이 어려울 때만 진행합니다.

이 순서 자체가 판단의 우선순위였습니다.

### 자동화 적용: RPA와 n8n의 선택 기준

자동화 분류로 넘어온 요청은 다시 두 가지 경로로 나뉘었습니다.
API 간 연결이 가능한 업무와, 화면 조작이 필수인 업무입니다.

#### 도구 선택의 기준: API가 있는가, 없는가

자동화 도구 선택의 첫 번째 질문은 단순했습니다.
대상 시스템이 API를 제공하는가, 아닌가입니다.

API가 있는 경우에는 n8n을 우선했습니다.
n8n은 오픈소스 워크플로우 자동화 도구로, 시스템 간 데이터를 API 기반으로 연결합니다.
내부망 서버에 설치해 운영했고, 스케줄 기반으로 백그라운드에서 실행되는 구조였습니다.

실제로 운영한 n8n 워크플로우는 시스템 간 데이터 동기화가 중심이었습니다.
SaaS에서 데이터를 가져와 다른 시스템에 입력하거나, 만료된 데이터를 자동 삭제하거나, 신규·수정 건을 감지해 반영하는 작업이었습니다.
사람이 화면을 조작할 필요가 없고, API 호출만으로 완결되는 업무였습니다.

반면 API로 완결할 수 없는 업무는 RPA(Robotic Process Automation)로 처리했습니다.
ERP 화면에서 데이터를 다운로드하고 가공하거나, 여러 시스템의 화면을 순차적으로 조작해야 하는 업무가 여기에 해당합니다.
UiPath 기반으로 운영했고, Unattended Robot 3대와 Attended Robot 1대, 개발자 라이선스 3개, Orchestrator를 포함한 구성이었습니다.

정리하면 다음과 같은 기준이었습니다.

- **API 간 연결이 가능한 업무** → n8n 우선. 백그라운드 실행, 스케줄 기반 자동화
- **화면 조작이 필수인 업무** → RPA. ERP·레거시 시스템의 UI 기반 반복 작업 자동화

이 구분은 단순해 보이지만, 도구 선택의 기준을 명확히 하는 것만으로 자동화 검토 속도가 달라졌습니다.
"이 업무에 어떤 도구를 쓸지"를 매번 논의하는 대신, API 유무라는 단일 질문으로 경로가 갈렸습니다.

#### 접수에서 착수까지: 요청이 자동화로 들어오는 과정

자동화 요청은 구조화된 접수 양식을 거쳐 들어왔습니다.
요청자는 과제 개발 요청서를 통해 다음 정보를 제출했습니다.

- **AS-IS**: 현재 업무 프로세스와 문제점. 어떤 시스템에서 어떤 순서로 작업하는지, 어디에서 반복과 오류가 발생하는지를 서술합니다.
- **업무 상세**: 1회 소요 시간, 작업 주기(일간/주간/월간/비정기), 오류 발생 빈도, 업무 중요도, 관련 시스템, 마감 기한을 기입합니다.
- **개발 요청 내용**: 자동화하려는 업무의 상세 프로세스를 단계별로 기술합니다. Flowchart나 작업 순서를 첨부할 수도 있었습니다.
- **TO-BE**: 기대효과, 확장·추가 가능성, 업무 지속성(몇 년간 사용할 것인지), 연관 업무 담당자를 기입합니다.

이 양식은 두 가지 목적을 가지고 있었습니다.
첫째, 요청자가 자신의 업무를 구조화하도록 유도하는 것이었습니다.
"현 업무 프로세스 및 문제점"을 서술하려면 단순히 "이것 좀 자동화해 주세요"로는 작성할 수 없었습니다.
어떤 시스템에 로그인해서, 어떤 데이터를 다운로드하고, 어떤 가공을 거쳐, 누구에게 전달하는지를 단계별로 정리해야 했습니다.

둘째, ROI 산정에 필요한 데이터를 수집하는 것이었습니다.
1회 소요 시간, 작업 주기, 오류 발생 빈도, 업무 중요도는 모두 ROI 점수 계산의 입력값이었습니다.

접수된 요청은 바로 개발로 넘어가지 않고 인터뷰를 먼저 진행했습니다.
업무 설계서에 없는 내용이 인터뷰에서 나오는 경우가 많았고, 프로세스 확장이 필요한 경우도 있었기 때문입니다.
인터뷰 후에 공수를 산정했고, 비용이 발생하는 경우 해당 내용을 안내한 뒤 진행 여부를 확인받았습니다.

현업과의 소통에도 기한을 정했습니다.
협의사항에 대한 답변 기한은 2일로 설정했고, 답변이 없으면 재통보 후 하루를 더 부여했습니다.
그래도 답변이 없으면 후순위로 밀렸습니다.
이 기준은 RPA 특성상 CSR(Change Service Request)이 등록돼도 바로 개발이 불가능한 구조 때문에 필요했습니다.
과제가 대기열에 머무는 시간을 줄이기 위한 운영 규칙이었습니다.

#### ROI 산정: 과제 선정의 정량 기준

모든 자동화 요청이 개발로 이어지지는 않았습니다.
과제 선정 기준은 ROI(Return on Investment) 점수였고, 이 점수를 산정하는 체계가 별도로 존재했습니다.

ROI 산정은 두 가지 관점으로 나뉘었습니다.
개발자 입장의 평가와 요청자 입장의 평가입니다.

**개발자 입장의 평가 항목**은 다섯 가지였습니다.

- **업무 복잡성 + 시스템 통합 난이도** (가중치 30): 케이스에 따른 다른 처리 방식이 포함되는지, 여러 시스템과의 연동이 필요한지를 평가합니다. 매우 복잡(1.0)에서 매우 단순(0.2)까지 5단계로 채점합니다. 업무가 복잡할수록, 여러 시스템과 연동될수록 ROI가 높아졌습니다. 수작업으로 처리하기 어려운 업무일수록 자동화의 가치가 크다는 판단이었습니다.
- **프로그램 개수** (가중치 5): 자동화에 관여하는 프로그램이 5개 이상이면 최고점, 1개이면 최저점입니다.
- **데이터 업데이트 유무** (가중치 25): 사이트 업로드나 데이터 수정이 포함되는지를 봅니다. 단순 조회가 아니라 데이터를 변경하는 업무일수록 자동화의 필요성이 높았습니다.
- **마감시간 제한** (가중치 25): 정해진 스케줄 시간 외에도 하루 안에 처리돼야 하면 "보통"으로, 매우 촉박한 마감이 있으면 최고점입니다. 시간 여유가 없는 업무는 문제 발생 시 사람이 개입할 시간이 부족하므로 자동화가 더 필요했습니다.
- **확장 가능성** (가중치 15): 계열사 확장이나 유사 업무로의 확장이 가능한지를 봅니다. 한 부서에서 자동화한 프로세스가 다른 법인에도 적용 가능하면 ROI가 올라갔습니다.

**요청자 입장의 평가 항목**은 여섯 가지였습니다.

- **업무 작업 시간** (가중치 35): 1회 기준 소요 시간입니다. 4시간 이상이면 최고점, 1시간 미만이면 최저점입니다. 가중치가 가장 높았습니다. 한 번 수행하는 데 시간이 많이 걸리는 업무일수록 자동화 효과가 컸기 때문입니다.
- **업무 작업 횟수/주기** (가중치 30): 시간 단위 반복이면 최고점, 연/분기 단위면 최저점입니다. 비정기 업무는 월 평균 횟수로 환산했습니다. 같은 작업을 하는 사람이 여러 명이면 해당 값을 전부 더해서 산출했습니다.
- **오류 발생률** (가중치 20): 수작업에서 오류가 많이 발생하는 경우 자동화의 필요성이 높아집니다.
- **업무 중요도** (가중치 15): 회사의 핵심 목표 달성과 직결되는 업무이면 최고점, 지원 업무로 부수적 중요성이면 최저점입니다.
- **업무 관련자 수** (가중치 0): 작업 결과물을 활용하는 사람의 수입니다. 200명 이상부터 9명 이하까지 5단계로 나뉩니다. 가중치를 0으로 설정해 점수에는 반영하지 않았지만, 참고 지표로 기록했습니다.
- **예상 비용 절감 효과** (가중치 0): 마찬가지로 가중치 0이었습니다. 정량 추정이 어려운 항목이라 참고용으로만 남겼습니다.

각 항목은 5단계(1.0, 0.8, 0.6, 0.4, 0.2) 중 하나로 채점하고, 가중치를 곱한 뒤 합산합니다.
총점은 개발자 입장 점수와 요청자 입장 점수의 평균으로 산출했습니다.

#### ROI 점수와 난이도 판정

산출된 ROI 점수는 난이도 등급과 개발 기간으로 연결됐습니다.

- **80점 이상** (상): 개발 기간 7주 이상
- **70~80점 미만** (중상): 개발 기간 6~7주 이상
- **60~70점 미만** (중): 개발 기간 4~6주 이상
- **50~60점 미만** (중하): 개발 기간 3~4주 이상
- **35~50점 미만** (하): 개발 기간 2~3주 이상
- **35점 미만**: 개발 불가

35점 미만은 개발 자체를 진행하지 않았습니다.
단순 작업으로 ROI 기준에 부합하지 않는 경우였습니다.
이 경우 요청자에게 "업무 프로세스 확인 결과 단순 작업은 과제 선정 기준(ROI)에 부합하지 않아 개발 진행이 불가능합니다"라고 안내했습니다.
다만 선후 프로세스를 추가해 재신청하는 경로는 열어두었습니다.

이 기준이 있었기 때문에 자동화 과제의 수락과 거절이 담당자 개인의 판단이 아니라 정량 기준에 근거할 수 있었습니다.

#### ROI 산정 전 필수 확인 사항

ROI 점수를 산정하기 전에 먼저 통과해야 하는 필수 확인 항목이 있었습니다.
이 항목에 걸리면 ROI 산정 자체가 진행되지 않았습니다.

- **업무 처리방식**: 규칙 기반으로 이루어진 작업이어야 합니다. 사람의 판단이 필요한 경우 자동화가 불가능했습니다.
- **업무 표준 유무**: 프로세스가 표준화될 수 있어야 합니다. 매번 다른 방식으로 처리되는 업무는 자동화 대상이 아니었습니다.
- **업무/양식 변경**: 개발 기간 내에 프로세스가 변경되지 않아야 합니다. 시스템 변경이 예정되어 있을 경우 개발 요청 자체를 지양했습니다.
- **예외 케이스**: 예외 상황이 많으면 복잡도가 올라가 자동화가 불가능할 수 있었습니다.
- **중복 로그인**: 자동화 시 중복 로그인이 가능한지, 별도 계정 생성이 가능한지를 확인했습니다.
- **IE(Web) 사용 여부**: 인터넷 익스플로러로만 접속 가능한 시스템은 자동화가 불가능했습니다.

특히 ERP 시스템 변경은 자동화 운영에 직접적인 영향을 미쳤습니다.
ERP의 화면 구조가 바뀌면 기존에 개발해둔 RPA 프로세스를 전부 수정해야 했습니다.
Element ID 값 변경, 메뉴 구성 변경, 다운로드 방식 변경, 로그아웃 동작 변경 등이 실제로 발생했고, 이때마다 운영 중인 과제들을 일괄 점검하고 수정하는 작업이 필요했습니다.
이런 유지보수 비용까지 고려하면 "시스템 변경이 예정된 업무는 자동화를 지양한다"는 기준은 현실적인 판단이었습니다.

#### ROI가 낮은 요청은 어떻게 처리했는가

ROI 35점 미만으로 개발 불가 판정을 받은 요청, 또는 35~50점 사이의 낮은 점수 요청이 문제였습니다.
개별로는 자동화 투자 대비 효과가 작았지만, 그렇다고 무시할 수 있는 것도 아니었습니다.
현업에서는 분명히 불편을 느끼고 있었고, 유사한 요청이 반복되는 경우도 있었습니다.

저희가 선택한 방식은 이런 요청을 따로 모아두는 것이었습니다.
개별 요청으로는 자동화 기준을 넘지 못하지만, 여러 요청을 묶으면 하나의 기능으로 개발할 수 있는 경우가 있었습니다.

예를 들어, 부서 A에서 "ERP 데이터를 다운로드해서 메일로 보내는 작업을 자동화해달라"는 요청이 들어옵니다.
1회 소요 시간 30분, 월 2회 수행. ROI 기준 미달입니다.
비슷한 시기에 부서 B에서도 "ERP의 다른 데이터를 다운로드해서 가공 후 공유하는 작업"을 요청합니다.
역시 개별로는 ROI 미달입니다.

하지만 두 요청의 구조를 보면 공통점이 있습니다.
ERP에서 데이터를 추출하고, 가공하고, 전달하는 패턴입니다.
이때 개별 자동화가 아니라 "ERP 데이터 추출 및 배포"라는 공통 기능으로 묶어 개발하면 ROI가 달라졌습니다.

이 접근은 소프트웨어 설계에서 `읽기와 쓰기를 분리하는 패턴`과 유사한 발상이었습니다.
개별 요청을 있는 그대로 각각 처리하는 대신, 요청의 구조를 분석해서 공통 패턴을 추출하고, 그 패턴을 하나의 기능으로 만든 것입니다.
개별 요청(읽기)은 그대로 수집하되, 실제 구현(쓰기)은 묶어서 처리하는 구조였습니다.

이 방식이 작동하려면 두 가지가 필요했습니다.
첫째, ROI 미달 요청을 버리지 않고 축적하는 체계입니다. 요청이 거절되더라도 기록은 남겨두었습니다.
둘째, 축적된 요청에서 공통 패턴을 발견하는 분석 역량입니다. 이 역할을 RPA 담당자들이 수행했습니다.
요청 분석 과정에서 자동화 적합 과제를 직접 발굴하는 역할로 확장된 것이 여기서 작동했습니다.

#### 자동화 운영의 현실: 구축보다 테스트가 오래 걸렸습니다

RPA 개발에서 실제로 시간이 많이 걸린 단계는 구축이 아니라 테스트였습니다.
프로세스 구현 자체는 비교적 빠르게 진행됐지만, 변수 상황에 대한 테스트가 오래 걸렸습니다.
유닛 테스트와 풀 테스트를 모두 수행해야 했고, 실제 업무 데이터로 검증하는 과정이 필수였습니다.

운영 중인 과제의 종류도 다양했습니다.
ERP 기반 회계 처리(GRIR Clearing, 수출 부대비용, 지급전표 처리), B2C 주문 관련 프로세스(주문 취합, 납품 등록, 송장 등록, 구매 확정), 데이터 수집 및 보고(생산 실적 보고, 의약품 안전 관련 보고, 문헌 검색), 시스템 간 데이터 검증(BP 휴폐업 확인, 요양기관기호 검증) 등이 운영되고 있었습니다.

과제마다 PDD와 유사 BPMN 흐름도를 정리했고, 이 문서가 운영 과정에서의 이슈 추적과 프로세스 변경 대응의 기준이 됐습니다.

#### MS Power Automate 전환: 라이선스와 운영 효율의 판단

UiPath 기반으로 운영하던 RPA 과제 중 일부는 MS Power Automate로 전환을 진행했습니다.
전환 대상은 ROI 점수 등급에 따라 결정됐습니다.

- **상(80점 이상)**: 전환 대상 아님. 복잡도가 높아 UiPath 유지
- **중상(70~80점 미만)**: 1개 전환
- **중(60~70점 미만)**: 3개 전환
- **중하(50~60점 미만)**: 4개 전환
- **하(35~50점 미만)**: 5개 전환

복잡도가 낮은 과제일수록 MS Power Automate로 전환하는 비율이 높았습니다.
이 판단의 배경에는 라이선스 비용과 운영 효율이 있었습니다.
UiPath의 연간 라이선스 비용은 약 4,520만 원 수준이었고, 이 비용 대비 운영 과제의 복잡도가 낮은 경우에는 이미 조직에서 사용 중인 MS 라이선스를 활용하는 것이 효율적이었습니다.

전환 계획은 기존 3명 인력으로 4.5개월이던 것이 DNC 전담 개발자 지정으로 인해 2명 이상 인력으로 7개월로 변경됐습니다.
전환 과정에서도 각 과제의 ROI를 재산정하고, 전환 후에도 동일한 수준의 안정성이 유지되는지 검증하는 과정을 거쳤습니다.
