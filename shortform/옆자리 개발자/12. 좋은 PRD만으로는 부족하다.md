## 명세만으로 협업이 되지 않는다.

> 바이브 코딩에선 세션(session)이 더 맞는 말이지만 이해를 돕기위해 사이클(Cycle)로 설명하겠습니다. 

---

**좋은 PRD는 한 번의 사이클에서는 강력합니다. 하지만 사이클이 바뀌거나, 사람이 바뀌거나, 수정이 이어지면 품질과 맥락이 무너집니다. PRD만으로는 부족하고, Task와 Process 관리가 필요해집니다.**

---

### 한 사이클의 성과

이전 글에서 PRD를 작성하고 코딩 에이전트에게 전달했습니다. 결과는 확실히 좋아졌습니다.

스펙이 없을 때 코딩 에이전트가 스스로 판단했던 것들 "정렬 방식, 표시 건수, 에러 처리, 시크릿 키 관리" 이 모두 PRD에 의해 고정되었습니다. 추측이 줄었고, 결과가 기대값에 가까워졌습니다.

한 사이클 안에서는 구현 방향을 안정적으로 잡는 데 충분했습니다.

---

### 사이클이 바뀌면 무너집니다

그런데 현실에서는 한 사이클으로 끝나는 프로젝트가 드뭅니다.

오늘 검색 기능을 만들고, 내일 결과 표시를 수정하고, 다음 주에 에러 처리를 개선합니다. 때로는 다른 사람이 이어받아 작업합니다.

이때 문제가 생깁니다.

**맥락(Context)이 사라집니다.** 코딩 에이전트의 대화 기록은 사이클이 끝나면 초기화됩니다. 어제 왜 그렇게 만들었는지, 어떤 제약 때문에 그 방식을 선택했는지 기억하지 못합니다.

**다른 사람이 이어받기 어렵습니다.** PRD를 보면 "무엇을 만들어야 하는지"는 알 수 있습니다. 하지만 "지금 어디까지 했는지", "왜 이 방식을 선택했는지"는 PRD에 없습니다.

**AI가 매번 다른 방식으로 구현합니다.** 같은 PRD를 주어도 사이클이 바뀌면 파일 구조, 변수명, 코드 스타일이 달라질 수 있습니다. 이전 사이클의 결정이 축적되지 않기 때문입니다.

**어디까지 했는지 추적이 안 됩니다.** PRD에 여러 요구사항이 있어도, 지금 어디까지 끝난 건지 바로 드러나지 않습니다.

쉽게 비유하면, 어제 만든 요리를 오늘 이어서 고치려고 하는데 AI가 처음부터 다시 만들어버리는 것과 같습니다. 추적하기 위해선 모든 작업물을 살펴보는 방법이 있는데 비용이 많이 들고 제대로 이해 못 할수도 있습니다.

---

### PRD만으로는 여전히 부족합니다

PRD를 외부 지침 파일에 저장하면 사이클이 바뀌어도 기준이 유지됩니다. 하지만 여전히 빠진 것이 있습니다.

**지금 무엇을 해야 하는지 모를 수 있습니다.** PRD에 요구사항이 여러 개 있어도, "지금 뭘 먼저 만들어야 하는가"는 정해져 있지 않습니다.

**누가 어디까지 끝냈는지 알기 어렵습니다.** 지침은 기준이지 진행 상태가 아닙니다. Task 문서가 있어도 실제 체크와 갱신이 없으면 추적은 되지 않습니다.

**구현 순서가 꼬이면 문제가 커집니다.** 코딩 에이전트에게 한 번에 모든 것을 시키면 서버, 클라이언트, API 연동, 에러 처리를 동시에 만들려고 합니다. 하나라도 잘못되면 전체가 흔들립니다.

**중간 검증 없이 진행하면 잘못된 방향으로 커질 수 있습니다.** 첫 단계가 틀렸는데 마지막까지 진행하면, 되돌리기 어렵습니다.

레시피(지침)는 있는데 요리 순서(프로세스)가 없는 상태입니다.

---

### Task: 무엇을 해야 하는가

Task는 PRD의 요구사항을 작은 단위로 쪼갠 것입니다. 소개한 3-파일 시스템의 Task List에 해당합니다.

이 프로젝트를 예로 들면 이렇게 나눌 수 있습니다.

**Task 1: 로컬 서버에서 앱 실행.** 서버를 만들고 브라우저에서 빈 페이지를 확인합니다.

**Task 2: 검색 입력 UI 구현.** 입력창과 검색 버튼을 만들고, 빈 검색어 처리를 추가합니다.

**Task 3: 서버에서 네이버 API 프록시 호출.** .env에서 인증 정보를 읽고, API를 호출하고, 결과를 JSON으로 반환합니다.

**Task 4: 결과 목록 렌더링.** API 응답을 파싱하여 제목, 요약, 언론사, 날짜, 링크를 표시합니다.

**Task 5: 로딩/에러/빈 결과 상태 처리.** 세 가지 상태를 각각 구현합니다.

**Task 6: 최종 점검.** 수용 테스트(AT-01~AT-06)를 하나씩 확인합니다.

핵심은 "무엇을 만들지"를 작은 단위로 쪼개는 것입니다.

한 번에 전체를 만들어달라고 하면, 코딩 에이전트는 모든 것을 동시에 시도합니다. 쪼개서 순서대로 시키면, 각 단계의 결과를 확인한 뒤 다음으로 넘어갈 수 있습니다.

> 기준이 잘 마련되어 있다면 순서대로 작업하지 않고 병렬로 만들어서 효율을 뽑는 방법도 있습니다.

---

### Process: 어떤 순서로 검증하는가

Task가 "무엇을"이라면, Process는 "어떤 순서로, 어떻게 검증하며"입니다.

**Step 1: 요구사항 합의.** PRD의 내용을 확인합니다. 불명확한 것이 있으면 구현 전에 정리합니다.

**Step 2: 환경 검증.** .env 파일에 인증 정보가 있는지, 실행 환경이 준비되었는지 확인합니다.

**Step 3: 최소 기능 구현.** Task 1~4를 순서대로 진행합니다. 각 Task 완료 후 브라우저에서 확인합니다.

**Step 4: 상태 처리 추가.** Task 5를 진행합니다. 로딩, 에러, 빈 결과를 각각 테스트합니다.

**Step 5: 테스트 및 승인.** Task 6을 진행합니다. 수용 테스트를 하나씩 확인합니다.

핵심은 "어떤 순서로 검증하며 만들지"를 고정하는 것입니다.

구현 전에 환경을 확인하고, 구현 후에 테스트하고, 테스트를 통과한 뒤에 다음으로 넘어갑니다. 이 흐름이 고정되면, 누가 언제 작업하든 같은 순서로 진행됩니다.

---

### 세 가지의 역할은 다릅니다

여기서 PRD, Task, Process의 차이를 정리합니다.

**PRD**는 "무엇을, 어떤 기준으로 만들지"를 정의합니다. 외부 지침 파일에 저장하면 사이클이 바뀌어도 기준이 유지됩니다.

**Task**는 "지금 무엇을 해야 하는지"를 쪼개서 관리합니다. PRD의 요구사항을 실행 가능한 단위로 나눈 것입니다.

**Process**는 "어떤 순서와 검증 흐름으로 진행할지"를 관리합니다. 구현 전 확인, 구현 후 테스트, 테스트 후 다음 단계. 이 순서를 고정합니다.

이 세 가지는 각각 다른 질문에 답합니다. 하나만으로는 충분하지 않습니다.

> 이것을 이을려면 PRD에 "작업시작시 반드시 Process List를 확인해주세요." 와 같은 지침을 추가해두면 됩니다. 그렇다면 AI도 사람도 Process List를 보고 어디까지 작업했고 다음작업이 무엇인지 알 수 있습니다.

---

### 정리하며

좋은 PRD는 시작점입니다.

외부 지침은 기준점입니다.

Task와 Process 관리는 지속 가능성을 만드는 운영 장치입니다.

한 번의 사이클에서 좋은 결과를 얻는 것은 PRD만으로 가능합니다. 하지만 여러 사이클, 여러 사람, 여러 차례 수정을 거쳐도 품질을 유지하려면, 이 세 가지가 함께 있어야 합니다.

스펙을 쓰고, 지침을 만들고, Task와 Process를 관리하면 충분할까요? 다음 글에서는 이 모든 것을 "얼마나" 써야 하는지를 다룹니다. 많이 쓸수록 좋은 것은 아닙니다.
