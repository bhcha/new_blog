# 뉴스를 모아서 저장하기까지

> **옆자리 개발자** — 워크플로우 자동화와 바이브 코딩

---

**「첫 번째 워크플로우」에서 노드 2개로 네이버 뉴스를 가져왔습니다. 이번에는 그 워크플로우를 실무 수준으로 고도화합니다. 국내 뉴스와 해외 뉴스를 합치고, 정리하고, n8n 내부 DB에 저장합니다. 노드를 먼저 찾지 않습니다. 업무를 핵심단어로 분해하고, 핵심단어가 노드를 찾아줍니다.**

---

### 지금까지의 워크플로우

「첫 번째 워크플로우」에서 만든 네이버 뉴스 워크플로우는 노드 2개였습니다.

- Schedule Trigger — 매일 아침 실행
- HTTP Request — 네이버 뉴스 API 호출

결과는 아이템 1개 안에 기사 99건이 들어있는 상태였습니다.
가져오기만 했습니다. 정리도, 저장도 하지 않았습니다.

---

### 실무에서 필요한 것

회사에서 매일 아침 산업 뉴스를 관련 임직원에게 공유하는 업무가 있다고 가정합니다.

필요한 것을 나열합니다.

- 국내 뉴스(네이버)와 해외 뉴스(newsapi.org) 두 소스
- 네이버는 한 번에 최대 100건. 결과가 더 있으면 여러 번 가져와야 함
- newsapi.org는 한 번에 전부 가져올 수 있음
- 두 API의 응답 형태가 다름. 통일해야 합침
- 제목, 링크, 설명만 저장
- n8n 내부 DB에 저장

---

### 두 API의 응답을 비교합니다

네이버 뉴스 API의 응답입니다.

```json
{
  "total": 902889,
  "start": 1,
  "display": 99,
  "items": [
    {
      "title": "기사 제목 (<b>키워드</b> 포함)",
      "link": "https://...",
      "description": "기사 요약...",
      "pubDate": "Tue, 24 Mar 2026 23:00:00 +0900"
    }
  ]
}
```

newsapi.org의 응답입니다.

```json
{
  "totalResults": 794,
  "articles": [
    {
      "source": { "name": "The Verge" },
      "title": "기사 제목",
      "description": "기사 요약...",
      "url": "https://...",
      "publishedAt": "2026-03-24T13:02:13Z"
    }
  ]
}
```

다른 점이 보입니다.

- 기사 배열의 이름이 다릅니다. `items` vs `articles`
- 링크 필드명이 다릅니다. `link` vs `url`
- 출처 정보가 다릅니다. 네이버는 없고, newsapi는 `source.name`
- 네이버는 제목에 `<b>` HTML 태그가 섞여 있습니다

이 차이를 맞추는 것이 이 워크플로우의 핵심입니다.

---

### 업무를 핵심단어로 분해합니다

노드를 먼저 찾지 않습니다.
"이 작업은 어떤 핵심단어인가"를 먼저 묻습니다.

---

**① 매일 아침 시작한다**

"같은 시각에 다시 시작한다."

> [!NOTE]
> **반복**입니다. 시간 기반 반복.

반복은 Trigger 카테고리에 있습니다. → **Schedule Trigger**

---

**② 네이버 뉴스를 가져온다**

네이버 뉴스 API에 요청을 보내서 데이터를 읽어옵니다.
원본을 바꾸지 않습니다.

> [!NOTE]
> **질의**입니다. 읽기만 하고 원본은 그대로.

질의는 Core 카테고리의 HTTP Request 또는 Actions in an App에 있습니다. 네이버 전용 노드는 없으므로 → **HTTP Request (GET)**

API 인증에 필요한 `X-Naver-Client-Id`와 `X-Naver-Client-Secret`이 있습니다.

> [!NOTE]
> 크레덴셜은 **맥락**입니다. 요청 데이터에는 없지만, 요청이 처리되려면 반드시 필요한 배경 정보.

---

**③ 결과가 100건을 넘으면 나머지도 가져온다**

네이버 뉴스 API는 한 번에 최대 100건(`display=100`)만 반환합니다.
전체 결과가 300건이라면 세 번 호출해야 합니다.

- 첫 번째: `start=1`, `display=100` → 1~100번 기사
- 두 번째: `start=101`, `display=100` → 101~200번 기사
- 세 번째: `start=201`, `display=100` → 201~300번 기사

"남은 데이터가 있는 한 계속 가져온다."

> [!NOTE]
> **반복**입니다. 하지만 ①과는 다른 종류입니다.
> ①은 "매일 아침 다시" — 시간 기반 반복.
> ③은 "데이터가 남아있는 한 계속" — 조건 기반 반복.
> 카페에서 "주문이 남아있는 한 계속 만든다"가 바로 이것입니다.

그리고 `start` 값이 1에서 101로, 101에서 201로 바뀝니다.
"지금 어디까지 가져왔는가"를 추적하고 있습니다.

> [!NOTE]
> `start` 값은 **상태**입니다. 카페 주문의 `접수됨 → 제조중 → 완성됨`과 같은 구조.

HTTP Request 노드에는 페이지네이션 기능이 내장되어 있습니다.
또는 Loop Over Items 노드로 직접 반복을 만들 수도 있습니다.

---

**④ newsapi.org 뉴스를 가져온다**

newsapi.org는 한 번에 전체 결과를 반환합니다.
페이지네이션이 필요 없습니다.

> [!NOTE]
> **질의**입니다. ②와 같은 핵심단어, 같은 노드(HTTP Request GET).

②와 ④는 서로 의존하지 않습니다.
Schedule Trigger에서 두 HTTP Request로 동시에 연결할 수 있습니다.

> [!NOTE]
> **순차**가 아닌 병렬도 가능합니다. 앞의 결과를 기다릴 필요가 없으면, 동시에 실행됩니다.

---

**⑤ 각각 아이템으로 펼친다**

네이버 응답은 아이템 1개 안에 `items` 배열로 기사 99건이 들어있습니다.
newsapi 응답은 아이템 1개 안에 `articles` 배열로 기사 수백 건이 들어있습니다.

n8n의 대부분의 노드는 아이템 단위로 동작합니다.
기사를 하나씩 다루려면, 배열을 아이템으로 펼쳐야 합니다.

"1개 안에 99건" → "99개의 아이템"

> [!NOTE]
> **변환**입니다. 데이터의 구조를 바꾸는 것.

변환은 Data transformation 카테고리에 있습니다. → **Split Out**

네이버는 `items` 필드를, newsapi는 `articles` 필드를 지정합니다.

---

**⑥ 같은 형태로 맞춘다**

펼친 뒤 두 소스의 필드명을 비교합니다.

- 네이버: `title`, `link`, `description`
- newsapi: `title`, `url`, `description`

`title`과 `description`은 같습니다.
`link` vs `url`이 다릅니다. 출처 정보도 다릅니다.

필요한 필드만 남기고, 필드명을 통일합니다.

- `title` → `title` (네이버는 `<b>` 태그 제거 필요)
- `link` / `url` → `link`
- `description` → `description`
- 출처: 네이버는 `"네이버"` 고정값, newsapi는 `source.name` 사용

> [!NOTE]
> **변환**입니다. 필요한 값만 골라서 형태를 맞추는 것.
> 「흐름을 나누고 값을 고른다」에서 Edit Fields로 필드를 줄인 것과 같습니다. 여기서는 "서로 다른 구조를 하나로 통일하는 것"으로 확장됩니다.

Data transformation → **Edit Fields (Set)**

네이버 제목의 `<b>` 태그 제거는 표현식으로 처리합니다.

```
{{ $json.title.replace(/<\/?b>/g, '') }}
```

---

**⑦ 두 소스를 합친다**

네이버 뉴스와 newsapi 뉴스가 같은 형태가 되었습니다.
두 흐름을 하나로 합칩니다.

> [!NOTE]
> **변환**입니다. 여러 흐름을 하나로 모으는 것.

Data transformation → **Merge (Append 모드)**

Append 모드는 두 입력의 아이템을 순서대로 이어 붙입니다.
형태를 먼저 맞췄기 때문에 자연스럽게 합쳐집니다.

---

**⑧ n8n 내부 DB에 저장한다**

정리된 뉴스를 n8n의 Data Table에 저장합니다.

Data Table은 n8n 안에 내장된 저장소입니다.
외부 서비스(Google Sheets, Airtable 등)를 연결하지 않아도 됩니다.

테이블 구조는 다음과 같습니다.

- `title` — 기사 제목
- `link` — 기사 링크
- `description` — 기사 설명
- `source` — 출처 (네이버 / The Verge 등)

> [!NOTE]
> **명령**입니다. 데이터를 쓰는 것. 외부든 내부든, 저장은 상태를 바꾸는 행위입니다.

Core → **Data Table 노드**

---

**⑨ 에러가 나면 알린다**

API 호출이 실패할 수 있습니다. 크레덴셜 만료, 네트워크 에러, 응답 구조 변경.

> [!NOTE]
> 실행 상태(Succeeded/Failed)는 **상태**입니다. 에러 정보(어디서, 왜 실패했는가)는 **맥락**입니다.

Error Trigger 워크플로우를 연결합니다. 「에러를 다루는 법」에서 다뤘던 패턴입니다.

---

### 핵심단어로 노드를 찾는 법

①~⑨를 만들면서 한 가지 패턴이 보입니다.

매번 같은 순서였습니다.
"이 작업은 어떤 핵심단어인가?" → "그 카테고리에서 노드를 찾는다."

정리하면 이렇습니다.

**질의** (읽기만, 원본 변경 없음)
→ Actions in an App (전용 노드가 있으면) 또는 Core > HTTP Request

**명령** (쓰기, 외부/내부 상태 변경)
→ Actions in an App 또는 Core > Data Table

**변환** (형태를 바꾸거나, 값을 고르거나, 합치거나)
→ Data transformation (Split Out, Edit Fields, Merge, Aggregate, Code)

**선택** (조건에 따라 갈래를 나누기)
→ Flow (If, Switch)

**반복** (되풀이)
→ Trigger 카테고리 (Schedule Trigger — 시간 기반)
→ Flow (Loop Over Items — 조건 기반)

이것이 「n8n을 열면 보이는 것」에서 정리한 "4가지 분류 → UI 카테고리" 매핑의 실전 활용입니다.

> 이것이 개발자가 n8n을 공부 없이 익히는 이유입니다.
> "이건 질의고, 저건 변환이고" — 구조가 보이면 노드는 찾아집니다.

---

### 직접 만듭니다

①~⑨를 실제로 만들어봅니다.

---

### Step 1. Schedule Trigger

「첫 번째 워크플로우」에서 만든 것과 같습니다.

1. 새 워크플로우를 만듭니다.
2. **Schedule Trigger**를 추가합니다.
3. **Trigger Interval**: `Custom (Cron)`
4. **Expression**: `0 8 * * 1-5` — 매주 월~금 아침 8시

---

### Step 2. 네이버 뉴스 API 호출 (페이지네이션 포함)

Schedule Trigger에 **HTTP Request** 노드를 연결합니다.

**기본 설정**

- **Method**: GET
- **URL**: `https://openapi.naver.com/v1/search/news.json`
- **Query Parameters**:
  - `query`: 검색 키워드 (예: `바이오`)
  - `display`: `100`
  - `sort`: `date`
- **Headers**:
  - `X-Naver-Client-Id`: 발급받은 Client ID
  - `X-Naver-Client-Secret`: 발급받은 Client Secret

**페이지네이션 설정**

HTTP Request 노드의 **Pagination** 옵션을 엽니다.

- **Pagination Mode**: `By URL Parameter`
- **Parameter Name**: `start`
- **First Page Value**: `1`
- **Page Size**: `100`
- **Max Pages**: 필요에 따라 설정 (예: `10` → 최대 1000건)
- **Complete When**: 응답의 `items` 배열이 비어있으면 중단

이렇게 하면 n8n이 `start=1`, `start=101`, `start=201`... 을 자동으로 올리면서 반복 호출합니다.

> [!NOTE]
> **반복**을 n8n이 대신 해줍니다. start 값이라는 **상태**를 자동으로 관리합니다.

---

### Step 3. newsapi.org 호출

Schedule Trigger에 **두 번째 HTTP Request** 노드를 연결합니다.
Schedule Trigger에서 두 개의 화살표가 나갑니다.

- **Method**: GET
- **URL**: `https://newsapi.org/v2/everything`
- **Query Parameters**:
  - `q`: 검색 키워드 (예: `biotech`)
  - `sortBy`: `publishedAt`
  - `language`: `en`
- **Headers**:
  - `X-Api-Key`: 발급받은 API Key

newsapi.org는 한 번에 전체 결과를 반환합니다.
페이지네이션이 필요 없습니다.

---

### Step 4. 아이템으로 펼치기

네이버 HTTP Request 뒤에 **Split Out** 노드를 연결합니다.
- **Field To Split Out**: `items`

newsapi HTTP Request 뒤에도 **Split Out** 노드를 연결합니다.
- **Field To Split Out**: `articles`

이제 아이템 1개 안에 기사 목록이 들어있던 것이, 기사 하나가 아이템 하나가 됩니다.

---

### Step 5. 형태 통일

각 Split Out 뒤에 **Edit Fields (Set)** 노드를 연결합니다.

**네이버 쪽 Edit Fields**

- `title`: `{{ $json.title.replace(/<\/?b>/g, '') }}` — HTML 태그 제거
- `link`: `{{ $json.link }}`
- `description`: `{{ $json.description.replace(/<\/?b>/g, '') }}`
- `source`: `네이버`

**newsapi 쪽 Edit Fields**

- `title`: `{{ $json.title }}`
- `link`: `{{ $json.url }}` — `url`을 `link`로 통일
- `description`: `{{ $json.description }}`
- `source`: `{{ $json.source.name }}` — The Verge, BBC 등

두 쪽 모두 **Include Other Input Fields**를 `false`로 설정합니다.
`title`, `link`, `description`, `source` 4개만 남깁니다.

---

### Step 6. 합치기

두 Edit Fields 노드를 **Merge** 노드에 연결합니다.

- **Mode**: `Append`

네이버 뉴스와 newsapi 뉴스가 하나의 흐름으로 합쳐집니다.
형태가 같으므로 문제없이 이어 붙여집니다.

---

### Step 7. 내부 DB에 저장

Merge 뒤에 **Data Table** 노드를 연결합니다.

먼저 n8n에서 Data Table을 만듭니다.

1. 왼쪽 패널에서 **Data Tables** 탭으로 이동합니다.
2. 새 테이블을 만듭니다. 이름: `daily_news`
3. 필드를 추가합니다.
   - `title` (Text)
   - `link` (Text)
   - `description` (Text)
   - `source` (Text)

워크플로우로 돌아와서 Data Table 노드를 설정합니다.

- **Operation**: `Insert`
- **Data Table**: `daily_news`
- **Mapping Mode**: `Map Automatically` — 필드명이 같으면 자동 매핑됩니다

---

### Step 8. 에러 워크플로우 연결

별도의 워크플로우를 만듭니다.

1. **Error Trigger** 노드를 추가합니다.
2. 알림 노드를 연결합니다 (Discord, Slack, Email 등).
3. 뉴스 수집 워크플로우의 **Workflow Settings**에서 이 에러 워크플로우를 지정합니다.

「에러를 다루는 법」에서 다뤘던 패턴 그대로입니다.

---

### 완성된 워크플로우

```
Schedule Trigger (매일 아침 8시)
    ├── HTTP Request (네이버, 페이지네이션)
    │   └── Split Out (items)
    │       └── Edit Fields (title, link, description, source=네이버)
    │                                                          ├── Merge (Append)
    └── HTTP Request (newsapi.org)                             │       └── Data Table (저장)
        └── Split Out (articles)                               │
            └── Edit Fields (title, link, description, source) ┘
```

---

### 정리하며

「첫 번째 워크플로우」에서 노드 2개로 시작한 것이 여기까지 왔습니다.

그 과정에서 한 것을 돌아봅니다.

- 매일 아침 시작한다 → **반복** → Schedule Trigger
- 두 소스에서 데이터를 가져온다 → **질의** → HTTP Request
- 결과가 남아있는 한 계속 가져온다 → **반복** (조건 기반) → 페이지네이션
- 배열을 아이템으로 펼친다 → **변환** → Split Out
- 필드명을 통일한다 → **변환** → Edit Fields
- 두 흐름을 합친다 → **변환** → Merge
- DB에 저장한다 → **명령** → Data Table
- 에러를 감시한다 → **상태** → Error Trigger

노드를 먼저 찾지 않았습니다.
"이 작업은 질의인가 명령인가"를 먼저 물었고, 그 답이 노드 패널의 어디를 열어야 하는지 알려주었습니다.

이것이 소프트웨어적 사고입니다.
