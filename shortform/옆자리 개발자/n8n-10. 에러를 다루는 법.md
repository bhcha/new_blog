# 에러를 다루는 법

**워크플로우를 만들었는데 실행이 실패합니다. 노드 설정이 잘못되었을 수도 있고, 외부 서비스가 응답하지 않을 수도 있습니다. 에러가 발생했을 때 당황하지 않으려면, 어디서 문제가 생겼는지 찾는 방법과 에러를 미리 대비하는 구조를 알아야 합니다.**

이번 편에서는 잘 만든 워크플로우가 실패할 때 무엇을 봐야 하는지 다룹니다.
여기서 중심이 되는 것은 `입력 → 처리 → 출력`, 그리고 `상태 / 맥락`입니다.
에러를 막연한 실패로 보지 않고, 어느 구간에서 무엇이 달라졌는지 좁혀 가는 관점으로 보겠습니다.

---

### 실행 로그에서 문제를 찾는다

n8n은 워크플로우의 실행 이력을 추적합니다.

왼쪽 패널에서 **Executions**를 선택하면 실행 로그가 열립니다.
최근 실행의 시간, **상태**(Succeeded, Failed, Running), 실행 방식, 소요 시간이 목록으로 나옵니다.

실행 상태는 이 워크플로우가 지금 어떤 단계에 있는지를 알려주는 값입니다.

> [!NOTE]
> 카페 주문의 `접수됨 → 제조중 → 완성됨`이 **상태**였던 것처럼, 워크플로우의 실행도 `Running → Succeeded` 또는 `Running → Failed`라는 상태를 갖습니다.

<img src="https://raw.githubusercontent.com/bhcha/new_blog/main/shortform/images/explanation_workflowexecutions.png" alt="실행 목록 화면">

Failed 상태인 실행을 선택하면, 워크플로우가 읽기 전용 모드로 열립니다.
각 노드의 실행 결과를 하나씩 확인할 수 있습니다.

<img src="https://raw.githubusercontent.com/bhcha/new_blog/main/shortform/images/explanation_workflowexecutions_readonly.png" alt="읽기 전용 실행 화면">

### IPO로 문제 위치를 좁힌다

읽기 전용 모드에서 각 노드를 클릭하면, 그 노드의 INPUT과 OUTPUT을 볼 수 있습니다.

> [!NOTE]
> 떠올려 봅니다.

무언가 안 될 때, "입력이 잘못된 건지, 처리가 잘못된 건지, 출력이 안 나오는 건지"를 나눠서 볼 수 있다고 했습니다.

실행 로그에서 하는 일이 정확히 그것입니다.

에러가 발생한 노드를 찾습니다.
그 노드의 INPUT을 봅니다. 이전 노드에서 넘어온 데이터가 예상과 다른지 확인합니다.
INPUT이 정상이라면 처리 설정을 확인합니다. 노드의 파라미터가 잘못 되어있을 수 있습니다.
처리도 정상이라면 OUTPUT을 봅니다. 외부 서비스의 응답이 예상과 다를 수 있습니다.

입력 → 처리 → 출력.
이 구조를 알면 디버깅이 "전체를 다시 살펴보는 것"이 아니라 "세 구간 중 어디가 문제인지 좁히는 것"이 됩니다.

### 에러를 자동으로 감지하는 워크플로우

문제를 찾는 방법을 알았습니다.
하지만 매번 실행 로그를 직접 확인할 수는 없습니다.

n8n에서는 **Error Trigger** 노드를 사용해서 별도의 에러 처리 워크플로우를 만들 수 있습니다.

메인 워크플로우가 실패하면, 이 에러 워크플로우가 자동으로 실행됩니다.

구조는 이렇습니다.

`Error Trigger(에러 수신)` → `알림 노드(Slack, Email 등)` → `메시지 전송`

이것도 입력 → 처리 → 출력입니다.
에러 정보가 들어오고(입력), 알림 메시지로 가공하고(처리), Slack이나 이메일로 보냅니다(출력).
메인 워크플로우와는 별도의 워크플로우이지만, 동일한 IPO 구조를 갖습니다.

에러 워크플로우를 설정하는 순서는 다음과 같습니다.

1. 새 워크플로우를 만들고 **Error Trigger** 노드를 추가합니다.
2. Error Trigger 뒤에 알림을 보낼 노드를 연결합니다. Slack, Discord, Telegram, Gmail 등 원하는 플랫폼을 선택합니다.
3. 모니터링할 워크플로우의 **Workflow Settings**에서, 방금 만든 에러 워크플로우를 지정합니다.

하나의 에러 워크플로우를 여러 워크플로우에 공통으로 지정할 수 있습니다.

### 에러 데이터라는 맥락

Error Trigger가 수신하는 데이터에는 디버깅에 필요한 정보가 담겨 있습니다.

- `execution.id`: 실패한 실행의 고유 번호
- `execution.error.message`: 에러 메시지
- `execution.lastNodeExecuted`: 마지막으로 실행된 노드 이름
- `workflow.id`, `workflow.name`: 어떤 워크플로우에서 발생했는지

이 정보들은 **맥락**입니다.

> [!NOTE]
> 맥락을 "직접적인 입력에는 없지만, 처리에 영향을 주는 배경 정보"라고 설명했습니다.

에러 데이터도 같습니다.
"어떤 워크플로우의 몇 번째 실행에서, 어느 노드가 마지막이었고, 무슨 에러가 났는지."
이 배경 정보가 있어야 문제의 원인을 추적할 수 있습니다.

---

### 에러 워크플로우의 제약

에러 워크플로우를 사용할 때 알아두어야 할 점이 있습니다.

Error Trigger는 워크플로우가 자동 실행(트리거에 의한 실행)으로 실패했을 때만 작동합니다.
수동으로 테스트 실행할 때는 Error Trigger가 발동하지 않습니다.

워크플로우에 Error Trigger 노드가 포함되어 있으면, 별도로 발행(Activate)하지 않아도 에러 워크플로우로 동작합니다.
또한 Error Trigger가 있는 워크플로우는 기본적으로 자기 자신을 에러 워크플로우로 사용합니다.

### 의도적으로 실패를 만드는 Stop and Error

에러는 예상하지 못한 곳에서만 발생하는 것이 아닙니다.
데이터가 잘못된 것을 미리 알아차렸을 때, 의도적으로 워크플로우를 멈추고 싶을 때가 있습니다.

**Stop and Error** 노드가 이 역할을 합니다.
이 노드는 워크플로우를 강제로 실패시키는 **명령**입니다.

두 가지 방식으로 에러를 지정할 수 있습니다.

- **Error Message**: 직접 작성한 에러 메시지를 반환합니다.
- **Error Object**: 에러 객체를 반환합니다.

Stop and Error 노드는 워크플로우의 마지막 노드로만 사용할 수 있습니다.

외부 서비스에서 가져온 데이터가 예상과 다를 때 유용합니다.
JSON 형식이 잘못되었거나, 숫자여야 할 값이 숫자가 아니거나, 필수 값이 빠져 있을 때.
이런 데이터는 당장 워크플로우를 멈추지는 않지만, 뒤에서 더 큰 문제를 만들 수 있습니다.
문제를 알아차린 시점에서 바로 멈추는 것이 나중에 원인을 추적하는 것보다 낫습니다.

### 마치며

에러를 다루는 일은 세 가지입니다.

실행 로그에서 각 노드의 입력과 출력을 확인해서 문제 위치를 좁힙니다.
Error Trigger 워크플로우로 실패를 자동 감지하고 알림을 받습니다.
Stop and Error 노드로 잘못된 데이터를 미리 차단합니다.

에러가 발생했을 때 당황하지 않는 것.
그것은 "에러가 없는 워크플로우"를 만드는 것이 아니라, "에러가 발생해도 빠르게 찾고 대응하는 구조"를 만드는 것입니다.
