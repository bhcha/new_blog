**이전 글에서 Nathan의 주간 매출 보고를 6단계로 나누었습니다. 이번 글에서는 그중 첫 두 단계를 실제로 만듭니다. 데이터를 가져오고, 저장하는 것. 소프트웨어에서 가장 기본적인 두 동작입니다.**

이번 편에서는 소프트웨어의 가장 기본적인 두 동작을 봅니다.
밖에서 값을 읽어오는 것은 `질의`이고, 밖의 상태를 바꾸는 것은 `명령`입니다.
`입력 → 처리 → 출력`의 흐름도 노드가 연결되면서 어떻게 이어지는지 함께 보겠습니다.

### 데이터 창고에서 데이터를 가져온다

Nathan의 워크플로우에서 가장 먼저 할 일은 회사의 오래된 데이터 창고에서 데이터를 가져오는 것입니다.

이전 글에서 Hacker News 노드를 사용했습니다. 특정 서비스 전용으로 만들어진 노드였습니다. 하지만 모든 서비스에 전용 노드가 있는 것은 아닙니다. Nathan의 회사 데이터 창고처럼, 전용 노드가 없는 경우도 있습니다.

다행히 데이터 창고에는 API 엔드포인트가 있습니다. 이것만 있으면 **HTTP Request** 노드로 데이터를 가져올 수 있습니다.

> HTTP Request 노드는 n8n에서 가장 범용적인 노드입니다. 전용 노드가 없는 앱이나 서비스에서도 HTTP 요청을 보내 데이터를 가져올 수 있습니다.

---

### HTTP Request 노드를 추가한다

새 워크플로우를 만듭니다. 이름은 "Nathan's workflow"로 지정합니다.

캔버스에 HTTP Request 노드를 추가합니다. 노드 창이 열리면, 설정할 것들이 있습니다.

<img src="https://raw.githubusercontent.com/bhcha/new_blog/main/shortform/images/l1-c5-5-1-http-request-node.png" alt="HTTP Request 노드">

이 노드는 **크레덴셜**을 사용합니다.

크레덴셜은 사용자나 서비스를 식별하는 고유 정보입니다. 아이디와 비밀번호가 대표적입니다. 서비스에 따라 형태가 다릅니다.

> [!NOTE]
> **맥락**입니다. "이 요청을 보내는 주체가 누구인가"라는 배경 정보입니다. 요청 데이터(입력)에는 없지만, 요청이 처리되려면 반드시 필요한 정보입니다.

---

### HTTP Request 노드를 설정한다

Parameters에서 다음을 설정합니다.

**Method**: GET으로 설정합니다.

데이터를 읽기만 합니다. 원본을 바꾸지 않습니다.

> [!NOTE]
> **질의**입니다. 메뉴판에서 레시피를 확인하는 것처럼, 가져오기만 하고 원본은 그대로입니다.

**URL**: n8n 과정 가입 시 이메일로 받은 Dataset URL을 입력합니다.

**Send Headers**: 켭니다(true). Specify Headers에서 Using Fields Below를 선택합니다.

- Header Parameters > Name: `unique_id`
- Header Parameters > Value: 이메일로 받은 Unique ID

**Authentication**: Generic Credential Type을 선택합니다.

- Generic Auth Type: Header Auth를 선택합니다.
- Credential for Header Auth: Create new credential을 선택합니다.
- Credentials 창에서 Name에 이메일로 받은 Header Auth name을, Value에 Header Auth value를 입력합니다.
- Save를 눌러 저장합니다.

<img src="https://raw.githubusercontent.com/bhcha/new_blog/main/shortform/images/l1-c5-5-1-http-request-node-credentials.png" alt="HTTP Request 크레덴셜 설정">

> 크레덴셜 이름은 기본적으로 "\<노드명\> account" 형식입니다. 이름을 바꿀 수 있습니다. 앱, 유형, 용도를 알 수 있는 이름을 붙이면 나중에 관리하기 편합니다.

저장이 끝나면 Credentials 창을 닫고 HTTP Request 노드로 돌아옵니다.

### 데이터를 가져온다

HTTP Request 노드에서 **Execute step** 버튼을 누릅니다.

결과가 테이블 뷰로 나타납니다. Nathan의 회사 데이터 창고에서 가져온 데이터입니다.

<img src="https://raw.githubusercontent.com/bhcha/new_blog/main/shortform/images/l1-c5-5-1-http-request-node-window.png" alt="HTTP Request 노드 실행 결과">

30명의 고객 주문 정보가 5개 필드로 구성되어 있습니다.

- `orderID`: 주문 고유 번호
- `customerID`: 고객 고유 번호
- `employeeName`: 담당자 이름
- `orderPrice`: 주문 금액
- `orderStatus`: 주문 상태. `booked`(확정) 또는 `processing`(처리중)

30개의 레코드가 한꺼번에 들어왔습니다.

> [!NOTE]
> **배열**입니다. 순서가 있는 목록. 대기 중인 주문서 여러 장이 한 묶음으로 도착한 것과 같습니다.

### Airtable에 데이터를 저장한다

Airtable을 사용합니다. 스프레드시트처럼 생겼지만, 데이터베이스의 기능을 가진 서비스입니다.

> Airtable 대신 Google Sheets 등 다른 스프레드시트 노드를 사용해도 됩니다.

### Airtable 테이블을 준비한다

데이터를 넣으려면 먼저 받을 그릇을 만들어야 합니다.

1. Airtable 계정을 만듭니다.
2. Workspace에 새 base를 만들고, 이름을 "beginner course"로 지정합니다.

<img src="https://raw.githubusercontent.com/bhcha/new_blog/main/shortform/images/l1-c5-2-create-airtable-base.png" alt="Airtable base 생성">

3. 기본으로 생성된 Table 1을 설정합니다.

기본 필드(`Name`, `Notes`, `Assignee`, `Status`)는 우리 데이터와 맞지 않습니다. 중요한 점이 있습니다. **Airtable의 필드 이름은 n8n에서 가져온 데이터의 필드 이름과 정확히 일치해야 합니다.**

테이블을 다음과 같이 준비합니다.

- 테이블 이름을 **orders**로 변경합니다.
- 기본 생성된 빈 레코드 3개를 삭제합니다.
- `Notes`, `Assignee`, `Status` 필드를 삭제합니다.
- `Name` 필드를 `orderID`로 변경하고, 필드 유형을 **Number**로 설정합니다.
- 나머지 필드를 추가합니다.
  - `customerID` — Number
  - `employeeName` — Single line text
  - `orderPrice` — Number
  - `orderStatus` — Single line text

### Airtable 노드를 연결한다

n8n 캔버스로 돌아옵니다.

HTTP Request 노드 옆의 **+** 아이콘을 눌러 Airtable 노드를 추가합니다. 검색창에서 Airtable을 찾고, Record Actions에서 **Create a record**를 선택합니다.

Airtable 노드 창에서 다음을 설정합니다.

**Credential to connect with**: Create new credential을 선택합니다.

- Connect using: Access Token을 유지합니다.
- Access token: Airtable에서 발급한 토큰을 입력합니다. 토큰을 생성할 때 beginners course base에 대한 접근 권한을 추가합니다.
- 저장하고 Credential 창을 닫습니다.

여기서도 크레덴셜이 필요합니다. HTTP Request 때와 마찬가지로, **맥락**입니다. "이 데이터를 저장하려는 주체가 누구인가"를 Airtable이 알아야 합니다.

**Resource**: Record
**Operation**: Create. 새 레코드를 만드는 작업입니다.
**Base**: beginner course를 선택합니다.
**Table**: orders를 선택합니다.
**Mapping Column Mode**: Map automatically. 들어오는 데이터의 필드명과 Airtable의 필드명이 같으면 자동으로 매핑됩니다.

### 노드가 연결되면 IPO 체인이 된다

한 걸음 물러서서 지금까지 만든 것을 봅니다.

HTTP Request 노드가 데이터를 가져왔습니다(질의). 그 결과가 Airtable 노드의 입력이 되었습니다. Airtable 노드가 데이터를 저장했습니다(명령).

HTTP Request의 **출력**이 Airtable의 **입력**이 된 것입니다.

> [!NOTE]
> **입력 → 처리 → 출력** 구조가 노드 하나에서만 일어나는 것이 아닙니다. 노드가 연결되면 앞 노드의 출력이 뒤 노드의 입력이 됩니다. 이것이 IPO 체인입니다. 워크플로우 전체가 하나의 큰 입력 → 처리 → 출력이 됩니다.
