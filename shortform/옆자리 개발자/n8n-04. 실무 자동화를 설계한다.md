# 실무 자동화를 설계한다


**반복되는 업무를 자동화하려면, 먼저 그 업무를 단계로 나눠야 합니다. n8n 공식 과정의 실무 시나리오를 따라가며, 자동화 설계가 어떻게 이루어지는지 살펴봅니다. 이전 글에서 다뤘던 핵심 단어들이 실제 업무에서 어떻게 보이는지 확인합니다.**

이번 편에서는 핵심 단어를 화면이 아니라 업무 설계의 관점에서 다시 봅니다.
자동화는 결국 일을 `질의`, `변환`, `명령`, `선택`, `반복`의 단계로 나누는 작업입니다.
특히 이번 편에서는 `상태`와 `맥락`이 왜 설계의 기준이 되는지 중심으로 보겠습니다.

---

### Nathan의 월요일 아침

Nathan은 ABCorp의 분석 매니저입니다.
매주 해야 하는 일이 있습니다.

회사의 레거시 데이터 웨어하우스에서 주문 데이터를 가져옵니다.
주문에는 두 가지 상태가 있습니다. `Processing`(처리중)과 `Booked`(확정).

Booked 주문의 합계를 계산해서, 매주 월요일 회사 Discord에 공유합니다.
Processing 주문은 따로 정리해서, 영업 담당자가 후속 조치할 수 있도록 스프레드시트에 넣습니다.

수작업입니다.
숫자를 하나하나 확인하고, 합산하고, 복사해서 붙여넣습니다.
집중력이 흐트러지면 숫자가 틀립니다.
한번은 상사에게 계산 오류를 지적받은 적도 있습니다.

Nathan은 이 작업을 자동화하고 싶습니다.

하지만 자동화 설계는 단계를 나누는 일에서 끝나지 않습니다.
그 단계가 정말 고정 가능한 규칙인지도 확인해야 합니다.
규칙이 자주 바뀌거나 사람의 판단이 계속 필요하다면, 자동화보다 정리 작업이 먼저일 수 있습니다.

---

### 자동화의 시작은 단계를 나누는 것입니다

Nathan이 매주 하는 일을 정리하면 여섯 단계입니다.

**1단계. 데이터 가져오기**
데이터 웨어하우스의 API에서 주문 데이터(주문 ID, 상태, 금액, 담당자)를 가져옵니다.
n8n에서는 HTTP Request 노드로 처리합니다.

> [!NOTE]
> **질의**입니다.
> 데이터를 읽어올 뿐, 원본이 바뀌지 않습니다.

**2단계. 주문 상태별 필터링**
가져온 주문을 상태에 따라 나눕니다.
`Booked`인 것과 `Processing`인 것을 분리합니다.
n8n에서는 If 노드로 처리합니다.

> [!NOTE]
> **선택**입니다.
> 조건에 따라 흐름이 갈라집니다.

**3단계. Booked 합계 계산**
Booked 주문의 금액을 모두 더합니다.
n8n에서는 Code 노드로 처리합니다.

> [!NOTE]
> **변환**입니다.
> 여러 건의 주문 금액이라는 재료가, 하나의 합계라는 결과물로 바뀝니다.

**4단계. 팀 알림**
계산된 합계를 회사 Discord 채널에 메시지로 보냅니다.
n8n에서는 Discord 노드로 처리합니다.

> [!NOTE]
> **명령**입니다.
> Discord라는 외부 서비스에 메시지를 보내는 것이니, 외부의 상태를 바꾸는 행위입니다.

**5단계. Processing 주문 저장**
Processing 상태의 주문을 Airtable에 저장합니다.
영업 담당자가 후속 조치할 수 있도록 정리하는 것입니다.
n8n에서는 Airtable 노드로 처리합니다.

> [!NOTE]
> 이것도 **명령**입니다.
> 외부 서비스(Airtable)에 데이터를 쓰는 것이니까요.

**6단계. 매주 자동 실행**
이 전체 흐름을 매주 월요일 아침에 자동으로 실행합니다.
n8n에서는 Schedule Trigger 노드로 처리합니다.

> [!NOTE]
> **반복**입니다.
> "매주 월요일"이라는 조건이 만족될 때마다 같은 흐름을 되풀이합니다.

---

### 전체 흐름은 순차입니다

여섯 단계를 이어 붙이면 하나의 워크플로우가 됩니다.

`Schedule → HTTP Request → If → Code(합계) → Discord`
`                              → Airtable`

앞의 것이 끝나야 다음으로 넘어갑니다.

If 노드에서 흐름이 두 갈래로 나뉘지만, 각 갈래 안에서는 순서대로 진행됩니다.
전체를 보면 순차이고, 가운데에 선택이 있는 구조입니다.

---

### 상태와 맥락이 보입니다

Nathan의 워크플로우에는 **상태**가 있습니다.

주문의 `orderStatus`가 그것입니다.
`Processing`이냐 `Booked`이냐에 따라 다음 행동이 달라집니다.
If 노드가 이 상태를 보고 흐름을 나눕니다.

그런데 한 가지 더 있습니다.

Nathan은 왜 Booked 합계를 Discord에 보내고, Processing 주문을 Airtable에 넣는 걸까요.
"Booked는 확정이니 팀 전체가 알아야 하고, Processing은 영업 담당자가 후속 조치해야 한다."
이것은 Nathan이 알고 있는 **비즈니스 규칙**입니다.

워크플로우 어디에도 적혀 있지 않습니다.
하지만 이 규칙이 워크플로우의 구조를 결정했습니다.
즉, 값 자체가 아니라 값을 어떻게 다뤄야 하는지 정하는 배경입니다.
이런것들을 맥락 이라고 합니다.

주문서에는 없었지만 처리 방식을 바꿨던 "단골 손님의 선호도"와 같습니다.
자동화를 설계할 때 가장 먼저 해야 할 일은, 이런 맥락을 명확하게 꺼내놓는 것입니다.

---

### 정리하며

Nathan의 월요일 아침 업무를 여섯 단계로 나누었습니다.

질의로 데이터를 가져오고, 선택으로 상태에 따라 나누고, 변환으로 합계를 계산하고, 명령으로 알림을 보내고 저장합니다.
이 전체가 순차로 흐르고, 반복으로 매주 실행됩니다.

주문 상태(`Processing` / `Booked`)는 흐름을 나누는 상태이고, "왜 이렇게 나누는가"는 워크플로우에는 없는 맥락입니다.

핵심 단어들이, 실무 자동화에서도 그대로 보입니다.
소프트웨어의 구조를 알면 자동화 설계가 읽힙니다.

### 참고

**데이터 웨어하우스(Data Warehouse)**: 조직의 주요 업무 데이터(매출, 생산, 고객 등)를 한곳에 모아 관리하는 시스템입니다. Nathan의 회사에서는 자체 제작한 레거시 시스템이 이 역할을 하고 있습니다.

**API(Application Programming Interface)**: 소프트웨어끼리 데이터를 주고받는 통로입니다. Nathan의 데이터 웨어하우스는 CSV 내보내기가 없지만, API가 있어서 n8n이 데이터를 가져올 수 있습니다.

**Airtable**: 스프레드시트처럼 생겼지만 데이터베이스처럼 쓸 수 있는 클라우드 서비스입니다. Nathan은 영업팀이 후속 조치할 Processing 주문을 여기에 정리합니다.
