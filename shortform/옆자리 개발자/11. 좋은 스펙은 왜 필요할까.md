# 좋은 스펙은 왜 필요할까

## AI가 잘 구현하게 만드는 PRD 작성법

> **AI 시대, 소프트웨어를 이해하는 법** 시리즈

---

**「맥락에서 명세로」에서 PRD와 Task 관리의 개념을 소개했습니다. 이번 글부터 세 편에 걸쳐 실제로 적용해봅니다. 좋은 스펙을 쓰면 결과가 좋아지고(이번 글), 외부 지침과 Task/Process 관리로 여러 세션에서도 유지되지만(다음 글) 너무 과한 스펙은 오히려 해가 됩니다(세 번째 글). 세 편을 함께 읽으시면 균형 잡힌 관점을 얻을 수 있습니다.**

---

### 한 줄 요청, 열 가지 해석

바이브 코딩의 시작은 보통 이렇습니다.

> 기존 건물 3D구현은 더 복잡한 과정이 필요해 간단한 예시로 명세를 작성하는것 부터 먼저 진행하고 그 뒤에 다뤄보겠습니다.

"회사 뉴스 검색 페이지 만들어줘."

이 요청은 얼핏 쉬워 보입니다. 하지만 이 한 줄 안에는 정해지지 않은 것이 너무 많습니다.

- 뉴스만 검색하는가, 블로그나 카페도 포함하는가?
- 어떤 회사명을 받는가? 한 개인가, 여러 개인가?
- 최신순인가, 정확도순인가?
- 몇 건을 보여주는가?
- 에러가 나면 어떻게 보여주는가? 결과가 없으면?
- API 키는 어디서 읽는가?
- 어디서 실행하는가? 로컬인가, 서버인가?

사람이 이 요청을 받으면 "그게 무슨 뜻이에요?"라고 되물을 수 있습니다. 하지만 코딩 에이전트는 대체로 묻지 않고, 스스로 판단해서 만듭니다.

문제는 그 판단이 매번 다르다는 것입니다.

같은 요청을 두 번 보내면, 한 번은 터미널에서 동작하는 콘솔 앱이 나올 수 있습니다. 다른 한 번은 React로 만든 풀스택 앱이 나올 수도 있습니다. 둘 다 "회사 뉴스 검색 페이지"라는 요청에 대한 해석입니다.

> 실제로 이 글을 위해 코딩 에이전트에게 이 한 줄만 전달하고 구현을 시켰습니다. 어떤 선택을 했는지, 어떤 것을 빠뜨렸는지는 이 글 마지막 부록에서 다룹니다.

---

### 좋은 스펙에 정답은 없습니다

여기서 "스펙을 잘 써야 한다"는 이야기를 하려고 합니다. 하지만 먼저 짚을 것이 있습니다.

좋은 스펙의 정답은 없습니다.

다만, AI를 위한 스펙작성이라면 개발자들 사이에서 합의되고 있는 방향은 있습니다. "모호함을 줄이는 문서가 결과를 좋게 만든다"는 것입니다. 하지만 어떤 형식으로, 어떤 항목을 넣어야 하는지는 프로젝트마다, 도구마다, 사람마다, 상황마다 다릅니다.

이 글에서 소개하는 PRD(Product Requirements Document) 구조는 여러 방법 중 하나입니다. 정답 템플릿이 아니라, 참조해야될 문서에 가깝습니다.

---

### 좋은 PRD의 최소 구조

「맥락에서 명세로」에서 PRD의 개념을 소개할 때, Ryan Carson의 3-파일 시스템을 바탕으로 설명했습니다. 이번에는 실제로 써봅니다.

잘 작동하는 스펙에는 공통된 항목이 있습니다. 무거운 문서가 아니어도 괜찮습니다. 아래 구조만 정리해도 결과가 달라집니다.

**Background & Problem** — 왜 이것을 만드는가. 배경과 문제를 한두 문장으로 정리합니다. 이것이 없으면 코딩 에이전트는 "무엇을 해결하려는지" 모른 채 코드를 만듭니다.

**Requirement (비개발자 시각)** — 무엇이 있어야 하는가. 비개발자의 시선에 해당합니다. 검색 대상은 뉴스만, 정렬은 최신순, 10건 표시처럼 비즈니스 관점에서 필요한 것을 정의합니다. "무엇을 만들지"를 정하는 영역입니다.

**Implementation Detail (개발자 시각)** — 어떻게 만들 것인가. 개발자의 시선에 해당합니다. 기술 스택, 서버 구조, API 호출 방식, 인증 처리 방법처럼 구현 방향을 정합니다. "어떻게 만들지"를 좁히는 영역입니다.

**Constraint (제약조건 — MUST NOT)** — 무엇을 하지 않아야 하는가. 이 항목이 특히 중요합니다. 코딩 에이전트는 제약이 없으면 가능한 모든 것을 시도합니다. "하지 말아야 할 것"을 명시하면 불필요한 구현이 줄어듭니다.

**Acceptance Tests (공통 — 비기능)** — 어떻게 확인하는가. "이것을 했을 때 이렇게 되면 완료"라는 기준입니다. 성공 케이스뿐 아니라 실패 케이스도 포함합니다. 비개발자와 개발자 모두가 같은 기준으로 완료를 판단합니다.

**SPEC (기본 스펙)** — 구현에 필요한 기술 명세입니다. API 엔드포인트, 요청 파라미터, 응답 필드처럼 코딩 에이전트가 바로 사용할 수 있는 정보를 담습니다.

이 정보들은 프로젝트 매순간마다 참조해야하는 지침만 있는 것이 좋습니다.

---

### AI에게 좋은 PRD는 사람이 읽기도 좋습니다

이 구조는 Requirement는 비개발자가, Implementation Detail은 개발자가 주로 채우지만, 완성된 PRD는 누구든 읽을 수 있다고 볼 수 있습니다.

"왜 만드는가, 무엇이 필요한가, 어떻게 만들 것인가, 무엇을 하지 않을 것인가, 어떻게 확인하는가."

이것은 사람에게 일을 맡길 때도 필요한 질문입니다. 팀원에게 업무를 전달할 때, 외주 개발사에 요구사항을 보낼 때, 상사에게 기획을 보고할 때. 배경, 요구사항, 구현 방향, 제약이 빠져 있으면 사람도 되묻거나 추측합니다.

AI에게 좋은 스펙은 사람이 읽어도 좋은 문서입니다. 모호함을 줄이는 문서는 누가 읽든 같은 그림을 그릴 수 있게 합니다.

### 같은 요청을 스펙으로 바꾸면

앞에서 "회사 뉴스 검색 페이지 만들어줘"라는 요청이 모호하다고 했습니다. 이것을 PRD로 바꿔보겠습니다.

**Background & Problem**
회사 관련 뉴스를 확인하려면 매번 포털에 접속해서 검색해야 합니다. 반복적인 작업이고, 검색 결과에서 필요한 정보만 골라보기 어렵습니다. 회사명을 입력하면 관련 뉴스를 한 화면에서 빠르게 확인할 수 있는 웹 페이지를 만듭니다.

**Requirement (비개발자 시각)**
- 검색 대상은 뉴스만 사용한다
- 기본 정렬은 최신순이다
- 최신 기사 10건이 표시된다
- 검색어 입력창이 있다
- 검색 버튼이 있다
- 검색 결과 목록이 표시된다
- 각 결과에 제목, 요약, 언론사, 날짜, 원문 링크가 포함된다
- 로딩, 에러, 빈 결과 상태가 각각 표시된다

**Implementation Detail (개발자 시각)**
- 서버: Node.js 빌트인 http 모듈로 HTTP 서버 구현
- 클라이언트: 단일 HTML + CSS + JS 구성, 프레임워크 없음
- API: 네이버 뉴스 검색 API 사용
- 프록시: /api/news 엔드포인트에서 네이버 API를 서버 사이드로 호출하여 결과를 클라이언트에 전달
- 인증: .env 파일을 직접 파싱하여 CLIENT_ID, CLIENT_SECRET 읽기 (dotenv 패키지 미사용)
- API 응답의 HTML 엔티티 제거 후 표시
- 검색 결과 2초 이내 표시

**Constraint (제약조건 — MUST NOT)**
- 앱은 localhost:8080에서만 실행한다
- CLIENT_SECRET은 서버에서만 사용한다. 브라우저에 절대 노출하지 않는다
- 외부 빌드 도구(webpack, vite 등)를 사용하지 않는다
- npm 패키지 의존성 없이 구현한다
- 네이버 API는 브라우저에서 직접 호출할 수 없다. 서버를 통해 프록시한다

**Acceptance Tests (공통 — 비기능)**
- "삼성전자"를 검색하면 뉴스 결과가 표시된다
- 각 결과에 제목, 요약, 언론사, 날짜, 링크가 보인다
- 링크를 클릭하면 원문 기사로 이동한다
- 검색어 없이 검색 버튼을 클릭하면 안내 메시지가 표시된다
- 존재하지 않는 검색어로 검색하면 빈 결과 메시지가 표시된다
- 서버가 꺼진 상태에서 검색하면 에러 메시지가 표시된다

**SPEC (기본 스펙)**
- 실행 주소: localhost:8080
- 검색 API: GET https://openapi.naver.com/v1/search/news.json
- 요청 파라미터: query(검색어), display(10), sort(date)
- 인증 헤더: X-Naver-Client-Id, X-Naver-Client-Secret
- 프록시 엔드포인트: GET /api/news?query={검색어}
- 응답 필드: title, description, originallink, link, pubDate

한 줄이었던 요청이 구조화된 문서가 되었습니다.

달라진 것은 분량이 아닙니다. 모호했던 부분이 고정되었다는 것입니다. 비개발자 시각에서 "무엇이 필요한지"가 정리되었고, 개발자 시각에서 "어떻게 만들지"가 좁혀졌습니다. 제약조건으로 "하지 말아야 할 것"이 명시되었고, 수용 테스트로 "어떻게 확인할지"가 정해졌습니다.

> 이 PRD를 코딩 에이전트에게 전달한 결과는 부록에서 v1(스펙 없음)과 비교합니다.

### PRD를 저장하는 방법

PRD를 작성했습니다. 그런데 이 문서를 코딩 에이전트에게 어떻게 전달할까요?

대부분의 코딩 에이전트 도구는 프로젝트 폴더에 특정 이름의 파일을 두면 세션이 시작될 때 자동으로 읽습니다.

> Claude Code에서는 `CLAUDE.md`, Cursor에서는 `.cursorrules`, GitHub Copilot에서는 `.github/copilot-instructions.md`를 사용합니다. 이름은 다르지만 역할은 같습니다.

PRD를 이 파일에 저장하면, 세션이 바뀌어도 코딩 에이전트가 매번 같은 기준을 읽고 시작합니다. 대화 기록은 사라져도 문서는 남습니다.

---

### 정리하며

좋은 스펙은 "많이 쓰는 문서"가 아닙니다. "모호함을 줄이는 문서"입니다.

정해진 형식은 없습니다. 하지만 배경, 요구사항, 구현 방향, 제약, 수용 테스트, 스펙. 비개발자의 "무엇을"과 개발자의 "어떻게"가 한 문서에 담기면, 코딩 에이전트가 추측해야 할 영역이 줄어듭니다. 결과가 기대값에 가까워집니다.

그리고 이 문서는 AI만을 위한 것이 아닙니다. 사람이 읽어도 같은 그림을 그릴 수 있는 문서입니다. 이말은 사람과 일을 할때도 말들면 유용한 문서라는 뜻 입니다.

이 방법이 유일한 정답은 아닙니다. 하지만 결과를 예측 가능하게 만드는 방향은 같습니다.

> PRD를 잘쓰는 법을 참고하시면 좋습니다.

이제 잘 시키는 방법은 알았습니다. 그런데 이걸 오늘이 아니라 컴퓨터를 끄고 다음날 시작한다면, 다른 사람이 이어받아서 해야한다면, 여러 차례 수정을 거쳐도 유지하려면 어떻게 해야 할까요?

### 부록. 스펙 없이 만든 결과 vs 스펙으로 만든 결과

이 글을 위해 같은 프로젝트를 두 가지 방식으로 코딩 에이전트에게 시켰습니다.

**v1: 스펙 없이**
요청: "회사 뉴스 검색 페이지 만들어줘"
이 한 줄만 전달했습니다. 나머지는 모두 코딩 에이전트가 판단했습니다.

에이전트가 스스로 내린 판단들입니다.

- Express, axios, dotenv 세 개의 npm 패키지를 설치했습니다. node_modules 폴더에 78개의 디렉토리가 생겼습니다.
- 포트를 3000번으로 정했습니다. Express의 기본값입니다.
- "뉴스는 최신이 중요하다고 가정"하여 최신순 정렬을 선택했습니다.
- 10건 표시를 "적당한 기본값, 별도 지정 없으므로 임의 결정"이라고 기록했습니다.
- "회사"가 특정 회사 하나인지 아무 회사나 검색하는 것인지 — 이것을 "가장 큰 모호함"이라고 기록했습니다.
- 요청하지 않은 페이지네이션 기능(이전/다음 버튼, 페이지 번호)을 스스로 추가했습니다. 범위가 확장된 것입니다.
- 검색어 없이 검색 버튼을 누르면 안내 메시지 없이 그냥 무시합니다.

에이전트가 스스로 불확실하다고 기록한 것들입니다.

- 뉴스 소스가 네이버만인지, 구글 뉴스도 포함할지
- 필터나 정렬 옵션이 필요한지
- 결과 표시 방식이 카드형인지 리스트형인지
- 자동완성이나 북마크 같은 부가 기능의 필요 여부

**v2: PRD 기반**
요청: 이 글에서 소개한 PRD 전문을 전달했습니다.

PRD가 대신한 판단들입니다.

- Constraint의 빌드 도구 금지와 npm 패키지 금지에 의해 Node.js 내장 모듈만 사용했습니다. Express를 설치하지 않았고, node_modules가 생기지 않았습니다.
- Constraint의 실행 주소 제약에 의해 포트를 8080으로 확정했습니다. 고민할 필요가 없었습니다.
- Requirement의 최신순 정렬과 10건 표시에 의해 정렬과 건수가 확정되었습니다.
- Implementation Detail의 프록시 구조와 Constraint의 CORS 제약에 의해 서버 사이드 호출이 필수로 정해졌습니다.
- Acceptance Tests의 빈 검색어와 빈 결과 테스트에 의해 에러 처리가 빠지지 않았습니다.
- SPEC의 API 엔드포인트와 파라미터 정보에 의해 API 호출 코드가 정확하게 작성되었습니다.

v2 에이전트는 이렇게 기록했습니다. "PRD가 없었다면 Express + dotenv를 사용했을 것이고, 포트 3000을 썼을 것이고, 에러 처리를 빠뜨렸을 것이다." 실제로 v1이 그랬습니다.

같은 프로젝트인데 결과가 다릅니다. 차이를 만든 것은 코딩 에이전트의 능력이 아니라 요청의 품질입니다. 스펙은 결과를 더 예측 가능하게 만들었고 검증 기준도 더 분명하게 만들었다고 보는 편이 정확합니다.

특히 두 가지 차이가 눈에 띕니다.

**Constraint(제약)가 가장 큰 차이를 만들었습니다.** v1은 Express, axios, dotenv 세 개의 패키지를 설치하고 포트 3000을 사용했습니다. v2는 순수 Node.js로 포트 8080을 사용했고 외부 의존성이 0입니다. "하지 말아야 할 것"을 명시하는 것이 "해야 할 것"을 나열하는 것만큼, 어쩌면 더 중요합니다.

**스펙이 없으면 범위가 확장됩니다.** v1은 요청하지 않은 페이지네이션 기능을 스스로 추가했습니다. v2는 PRD에 정의된 범위 안에서만 구현했습니다. 스펙은 "무엇을 만들지"뿐 아니라 "어디까지 만들지"도 정합니다.
