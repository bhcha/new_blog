# LLM이 모르는 것을 알게 하려면

## Fine-tuning, RAG, Context-Window, Function Calling

> **AI 시대, 소프트웨어를 이해하는 법** 시리즈

> 원본: [https://sprawlingdev.substack.com/p/llm-6e5](https://sprawlingdev.substack.com/p/llm-6e5)

---

> 이 부분은 나눠서 설명하면 제가 아는것만 해도 엄청 길게 풀어야 하나 비개발자 분들을 위한 글인만큼 여러개 묶어 가볍게 훑어 봅니다. 그리고 저도 모르는게 많습니다.

**LLM은 사전에 학습한 도화지 안에서만 콘텐츠를 생성합니다. 도화지 밖의 정보는 모릅니다. 모르는 것을 알게 하는 방법은 크게 두 가지입니다. 도화지 자체를 바꾸거나, 도화지는 그대로 두고 정보를 넣어주는 것입니다.**

---

### LLM은 왜 “모르는 게” 생기는가

「LLM이란?」에서 LLM의 콘텐츠 생성을 도화지 위에 점을 찍는 것으로 비유했습니다. 도화지는 사전에 학습된 거대한 텍스트 모델이었습니다.

이 도화지에는 한계가 있습니다.

**학습 시점의 한계.** LLM은 특정 시점까지의 공개 데이터로 학습됩니다. 그 이후에 나온 정보는 도화지에 없습니다. 어제 발표된 뉴스, 오늘 변경된 법률, 올해 출시된 신제품. 학습 시점 이후의 세상은 모릅니다.

> 재미있는 일화가 있습니다. gemini cli가 나온지 몇일 지나 gemini에게 gemini cli 설치법을 알려달라고 얘기했는데 그런게 없다고 계속 우기는 겁니다. 이게 모델의 한계 입니다....재미..있죠?
>
> 요즘은 이런 경우 자동으로 웹 검색을 해서 보완해 주기도 합니다. 하지만 그것은 모델이 알게 된 것이 아니라, 뒤에서 설명할 Function Calling으로 부족한 정보를 가져온 것입니다.

**비공개 정보의 한계.** 회사 내부 문서, 사내 시스템 데이터, 비공개 매뉴얼. 이런 것들은 공개된 적이 없으므로 학습 데이터에 포함되지 않습니다. 도화지에 처음부터 존재하지 않는 정보입니다.

문제는, 모르는 것도 그럴듯하게 생성한다는 점입니다.

LLM은 “모릅니다”라고 답하지 않습니다. 학습된 패턴을 따라 그럴듯한 콘텐츠를 만들어 냅니다. 존재하지 않는 논문을 인용하거나, 없는 사내 규정을 정확한 것처럼 제시합니다. 이것을 할루시네이션(Hallucination, 환각)이라고 합니다.

---

### 해결 방향은 크게 두 가지입니다

도화지 밖의 정보를 다루는 방법은 두 갈래로 나뉩니다.

**도화지 자체를 바꾸는 법.** 모델에 추가 학습을 시켜서 도화지의 질감, 색감, 기본 방향을 변경합니다. 이것이 파인튜닝(Fine-tuning)입니다.

**도화지는 그대로, 정보만 넣어주는 법.** 모델은 건드리지 않습니다. 대신 필요한 정보를 요청과 함께 전달합니다. 컨텍스트 윈도우에 직접 넣거나, 검색으로 찾아서 넣거나, 외부 시스템에서 실시간으로 가져옵니다.

---

### 파인튜닝 — 도화지 자체를 바꾸는 것

파인튜닝은 이미 학습을 마친 LLM에 추가 학습을 시키는 것입니다. 특정 분야의 데이터, 원하는 응답 방식, 특정 어조로 학습하면 모델의 내부 파라미터[1](https://sprawlingdev.substack.com/p/llm-6e5#footnote-1-188903825)가 변경됩니다.

도화지의 질감이 바뀌는 것입니다. 같은 요청을 해도 이전과 다른 콘텐츠가 생성됩니다.

[![Pasted image 20260223225723.png](images/llm-6e5/img_00_6524c5e5.png "Pasted image 20260223225723.png")](https://substackcdn.com/image/fetch/$s_!Dqvp!,f_auto,q_auto:good,fl_progressive:steep/https%3A%2F%2Fsubstack-post-media.s3.amazonaws.com%2Fpublic%2Fimages%2F3dafdfdd-4d84-43f7-8cc8-eeb30cd30108_1798x524.png)

> 대표적인 사례가 ChatGPT입니다. GPT는 텍스트를 예측하는 모델이었지만, 대화형으로 파인튜닝한 결과가 ChatGPT입니다. 같은 기반 모델이지만 도화지가 바뀌었기 때문에, 같은 요청에도 전혀 다른 방식으로 응답합니다.

다만 도화지를 바꾸는 것은 되돌릴 수 없습니다.

---

### 컨텍스트 윈도우에 직접 넣기 — 가장 단순한 방법

가장 직관적인 방법입니다. LLM에게 요청할 때, 필요한 정보를 함께 붙여 넣는 것입니다. 문서 전문을 넣고 “문서 요약해줘” 라고 하는 것이 대표적인 예입니다.

클로드 프로젝트에 책을 하나 넣고 쓰는것도 컨텍스트 윈도우[2](https://sprawlingdev.substack.com/p/llm-6e5#footnote-2-188903825)를 사용하는 방법중 하나 입니다.

[![Pasted image 20260223231518.png](images/llm-6e5/img_01_39324f78.png "Pasted image 20260223231518.png")](https://substackcdn.com/image/fetch/$s_!vn-V!,f_auto,q_auto:good,fl_progressive:steep/https%3A%2F%2Fsubstack-post-media.s3.amazonaws.com%2Fpublic%2Fimages%2Fa7806d55-dad4-41e6-9028-1d996384cc4b_1728x1990.png)

하지만 한계가 분명합니다.

**컨텍스트 윈도우 크기의 제한.** 현재 주요 모델은 128K~200K 토큰[3](https://sprawlingdev.substack.com/p/llm-6e5#footnote-3-188903825)을 지원합니다. 영어 기준 책 한 권 정도입니다. 하지만 한국어는 2~3배 토큰을 소모하므로 실질 범위는 더 좁습니다. 문서 10개를 한꺼번에 넣으면 금방 한계에 닿습니다.

**넣는 양이 늘면 비용과 시간이 증가합니다.** 토큰이 많을수록 과금이 올라가고, 응답이 느려집니다.

**중간 정보를 놓칩니다.** 연구에 따르면 LLM은 앞부분과 뒷부분은 잘 활용하지만, 중간에 있는 정보는 놓치는 경향이 있습니다.

> 이건 아무리 생각해도 도화지로 비유 못하겠습니다.

---

### RAG — 필요한 것만 찾아서 넣기

직접 넣기의 한계는 명확했습니다. 모든 정보를 넣을 수 없고, 많이 넣으면 비효율적입니다.

RAG(Retrieval-Augmented Generation)는 이 문제를 검색으로 해결합니다. 모든 정보를 넣는 대신, 요청에 관련된 정보만 찾아서 넣어줍니다.

동작 방식은 단순합니다.

요청이 들어오면, 별도의 데이터베이스에서 관련 문서를 먼저 검색합니다. 검색된 문서를 요청과 함께 컨텍스트 윈도우에 넣습니다. LLM이 그 자료를 참조하며 콘텐츠를 생성합니다.

도화지 비유로 보면, 도화지는 그대로입니다. 다만 요청할 때마다 관련 참고 자료를 자동으로 찾아서 옆에 놓아주는 것입니다.

[![Pasted image 20260223231018.png](images/llm-6e5/img_02_60180eb4.png "Pasted image 20260223231018.png")](https://substackcdn.com/image/fetch/$s_!Uzvf!,f_auto,q_auto:good,fl_progressive:steep/https%3A%2F%2Fsubstack-post-media.s3.amazonaws.com%2Fpublic%2Fimages%2Ff72b8aac-d472-481e-89b9-07690760b24a_828x652.png)

RAG의 장점은 도화지를 건드리지 않는다는 것입니다. 데이터베이스의 문서를 업데이트하면 바로 최신 정보가 반영됩니다. 파인튜닝처럼 모델을 다시 학습시킬 필요가 없습니다.

다만 `참조하는 자료의 품질이 생성의 품질을 결정합니다.` 잘못된 문서가 검색되면, 잘못된 근거 위에서 그럴듯한 콘텐츠가 생성됩니다. 할루시네이션보다 위험할 수 있습니다. 근거가 있는 것처럼 보이기 때문입니다.

> RAG에도 한계가 있고 최적화 방법도 여러가지 입니다.

---

### Function Calling — 실시간으로 외부에서 가져오기

RAG는 미리 저장해 둔 문서에서 검색합니다. 하지만 실시간으로 변하는 데이터는 어떨까요. 지금 이 순간의 환율, 현재 재고 수량, 오늘의 주가.

이런 데이터는 저장해 두는 것이 아니라, 필요할 때 직접 가져와야 합니다.

Function Calling은 LLM이 외부 시스템의 기능을 호출할 수 있게 하는 방식입니다. LLM에게 “이런 기능을 사용할 수 있다”고 알려주면, 필요한 순간에 스스로 해당 기능을 호출합니다.

[![Pasted image 20260223231200.png](images/llm-6e5/img_03_42b2c876.png "Pasted image 20260223231200.png")](https://substackcdn.com/image/fetch/$s_!OvMS!,f_auto,q_auto:good,fl_progressive:steep/https%3A%2F%2Fsubstack-post-media.s3.amazonaws.com%2Fpublic%2Fimages%2Fdb44ad34-e090-4504-b466-c55c8f9249c0_1446x1936.png)

도화지 비유로 보면, 도화지 옆에 전화기를 놓아준 것과 같습니다. 참고 자료가 옆에 있는 것이 아니라, 필요할 때 전화를 걸어서 정보를 받아오는 것입니다.

> LLM이 직접 외부 시스템을 실행하는 것은 아닙니다. “이 기능을 호출하겠다”는 의도와 필요한 값을 생성하면, 중간의 시스템이 실제 호출을 수행하고 결과를 LLM에게 돌려줍니다. LLM은 그 결과를 참조하여 콘텐츠를 생성합니다.

RAG가 도서관에서 책을 찾아오는 것이라면, Function Calling은 담당자에게 전화를 걸어 지금의 상황을 확인하는 것입니다.

---

### 정리: 언제 뭘 써야 하나

LLM이 모르는 정보를 다루는 방법을 정리하겠습니다.

**컨텍스트 윈도우 직접 삽입.** 소량의 문서, 간단한 맥락 제공에 적합합니다. 시스템 프롬프트에 규칙이나 용어집을 넣는 것이 대표적입니다. 구축할 것이 없어 즉시 사용할 수 있지만, 양이 늘면 한계에 닿습니다.

**RAG.** 사내 문서, 매뉴얼, 과거 데이터처럼 축적된 정보를 활용할 때 적합합니다. 문서가 업데이트되면 바로 반영됩니다. 검색 시스템을 구축해야 하지만, 모델을 건드리지 않으므로 유연합니다.

**Function Calling.** 실시간 데이터가 필요할 때 적합합니다. 환율, 재고, 일정처럼 지금 이 순간의 값을 가져와야 할 때 사용합니다. 연결할 외부 시스템이 있어야 합니다.

**파인튜닝.** 모델의 행동 방식 자체를 바꿔야 할 때 적합합니다. 특정 어조, 특정 분야의 전문 용어, 일관된 응답 패턴이 필요할 때 고려합니다. 비용과 전문성이 필요하고, 되돌릴 수 없으므로 다른 방법으로 해결되지 않을 때 마지막으로 고려합니다.

대부분의 실무 상황에서는 직접 삽입이나 RAG로 해결됩니다. Function Calling은 실시간성이 필요할 때 추가합니다. 파인튜닝은 가장 마지막에 고려합니다.

`도화지를 바꾸기 전에, 옆에 놓아줄 수 있는 것부터 시도합니다.`

### 부록. 이것들을 조합해서 쓸 수 있나요?

실제로는 하나만 쓰는 경우가 드뭅니다.

예를 들어 사내 챗봇을 만든다면, 시스템 프롬프트에 회사 규칙과 어조를 넣고(직접 삽입), 사내 문서에서 관련 정보를 검색해서 참조하고(RAG), 인사 시스템에서 잔여 연차를 조회합니다(Function Calling).

각 방법은 배타적이지 않습니다. 필요에 따라 조합합니다.

다만 조합이 복잡해질수록 구축과 운영의 난이도가 올라갑니다. 처음부터 모든 것을 갖추려 하기보다, 가장 단순한 방법부터 시작해서 부족한 부분을 하나씩 보완하는 것이 현실적입니다.
