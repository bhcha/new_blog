# 자동으로 돌린다

**지금까지 만든 워크플로우는 Execute 버튼을 눌러야 실행되었습니다. 이번 글에서는 워크플로우를 자동으로 실행하고, 실행 결과를 확인하고, 워크플로우를 내보내고 가져오는 방법을 다룹니다.**

이번 편에서는 워크플로우를 한 번 만드는 데서 끝내지 않고, 계속 돌리는 방법을 봅니다.
중심이 되는 것은 `반복`, `상태`, `맥락`입니다.
언제 실행되는지, 지금 어떤 상태인지, 어떤 배경 설정이 동작을 바꾸는지를 함께 확인하겠습니다.

---

### Manual Trigger를 Schedule Trigger로 교체합니다

지금까지 워크플로우의 시작점은 Manual Trigger였습니다.
Execute Workflow 버튼을 누르면 실행됩니다.

하지만 Nathan은 매주 월요일 아침마다 이 버튼을 누르고 싶지 않습니다.
자동으로 실행되어야 합니다.

n8n에는 Schedule Trigger 노드가 있습니다.
정해진 날짜, 시간, 간격에 맞춰 워크플로우를 자동으로 시작하는 노드입니다.

Manual Trigger를 지우고, Schedule Trigger로 교체합니다.

---

### Schedule Trigger 설정

먼저 기존 Manual Trigger 노드를 삭제합니다.
노드를 선택한 뒤 휴지통 아이콘을 누르면 됩니다.

노드 패널에서 Schedule Trigger를 검색해서 추가합니다.
설정할 항목은 다섯 가지입니다.

- **Trigger Interval**: Weeks를 선택합니다
- **Weeks Between Triggers**: 1을 입력합니다
- **Trigger on weekdays**: Monday를 선택합니다 (기본값으로 Sunday가 있다면 제거합니다)
- **Trigger at Hour**: 9am을 선택합니다
- **Trigger at Minute**: 0을 입력합니다

설정을 마치면 Schedule Trigger를 HTTP Request 노드에 연결합니다.
화살표를 드래그해서 이어주면 됩니다.

<img src="https://raw.githubusercontent.com/bhcha/new_blog/main/shortform/images/l1-c5-5-7-schedule-trigger-node.png" alt="Schedule Trigger 노드 설정">

> [!NOTE]
> **입력 → 처리 → 출력**을 떠올려 보십시오.

Manual Trigger에서 Schedule Trigger로 바꾼 것은, 워크플로우의 **입력** 방식을 바꾼 것입니다.
사람이 버튼을 누르는 입력에서, 시간 조건에 의한 입력으로 바뀌었습니다.
처리와 출력은 그대로입니다.

> [!NOTE]
> "매주 월요일 9시"라는 설정은 **반복**입니다.

대기 주문이 남아있는 한 계속 만들었던 바리스타처럼, 매주 월요일이 돌아올 때마다 같은 흐름을 되풀이합니다.

---

### 타임존 설정에 주의합니다

Schedule Trigger는 타임존에 따라 실행 시각이 달라집니다.

워크플로우의 타임존이 설정되어 있으면 그것을 따르고, 설정되어 있지 않으면 n8n 인스턴스의 기본 타임존을 따릅니다.
"매주 월요일 9시"가 한국 시간인지 미국 시간인지에 따라 결과가 달라지니, 반드시 확인해야 합니다.

타임존 설정은 뒤에서 다루는 Workflow Settings에서 할 수 있습니다.

자동 실행은 편합니다.
하지만 사람이 직접 누르지 않기 때문에 더 조심해야 합니다.
예를 들어 타임존이 잘못되면 월요일 오전 9시가 아니라 다른 시간에 실행될 수 있습니다.

---

### Publish — 워크플로우를 활성화합니다

Schedule Trigger를 설정했다고 바로 자동 실행되지는 않습니다.
워크플로우를 **Publish** 해야 합니다.

Publish는 에디터 상단 내비게이션에 있는 버튼입니다.
이것을 누르면 워크플로우가 활성화됩니다.

<img src="https://raw.githubusercontent.com/bhcha/new_blog/main/shortform/images/l1-c5-5-8-activated-workflow.png" alt="활성화된 워크플로우">

활성화된 워크플로우는 트리거 노드가 입력을 받거나 조건이 충족될 때마다 자동으로 실행됩니다.
새로 만든 워크플로우는 기본적으로 비활성 상태입니다.

> [!NOTE]
> **상태**입니다.

워크플로우 자체에 "활성/비활성"이라는 상태가 있고, 이 상태에 따라 다음 행동이 달라집니다.
비활성이면 아무 일도 일어나지 않고, 활성이면 매주 월요일 9시에 실행됩니다.
Publish 버튼을 누르는 것은 이 상태를 바꾸는 행위입니다.

---

### 실행 로그 — 워크플로우가 잘 돌았는지 확인합니다

워크플로우가 자동으로 실행되면, 결과를 직접 보지 못할 수 있습니다.
그래서 n8n은 실행 로그를 남깁니다.

실행(Execution)은 워크플로우가 처음 노드부터 마지막 노드까지 한 번 완료된 기록입니다.
실행 로그를 보면 워크플로우가 성공했는지, 실패했는지 알 수 있습니다.

확인하는 방법은 두 가지입니다.

- **워크플로우 단위**: 캔버스에서 Executions 탭으로 전환합니다
- **전체 인스턴스 단위**: Overview에서 Executions 탭을 선택합니다

실행 로그에는 다음 정보가 표시됩니다.

- **Name**: 워크플로우 이름
- **Started At**: 실행 시작 시각
- **Status**: 실행 상태 (Waiting, Running, Succeeded, Cancelled, Failed)
- **Execution ID**: 이 실행의 고유 ID

<img src="https://raw.githubusercontent.com/bhcha/new_blog/main/shortform/images/l1-c5-5-8-execution-list.png" alt="실행 목록">

워크플로우별로, 상태별로 필터링할 수 있습니다.

> [!NOTE]
> 각 실행의 Status는 **상태**입니다.

카페에서 주문의 상태가 `접수됨 → 제조중 → 완성됨`이었던 것처럼, 실행의 상태가 `Waiting → Running → Succeeded` 또는 `Failed`로 바뀝니다.
상태를 보면 다음 행동을 판단할 수 있습니다.

> [!NOTE]
> 만약 Failed가 보인다면, **입력 → 처리 → 출력**으로 문제 위치를 좁힐 수 있습니다.
입력이 잘못된 건지(API 주소가 바뀌었거나), 처리가 잘못된 건지(조건이 틀렸거나), 출력이 안 나오는 건지(외부 서비스가 응답하지 않거나).

그래서 자동화는 "돌아가게 만드는 것"만큼 "언제 어떻게 돌았는지 확인하는 것"도 중요합니다.

---

### Workflow Settings — 워크플로우의 배경 설정

에디터 오른쪽 상단의 점 세 개 메뉴에서 Settings를 선택하면 워크플로우 설정 화면이 나타납니다.

<img src="https://raw.githubusercontent.com/bhcha/new_blog/main/shortform/images/l1-c5-5-8-workflow-settings.png" alt="워크플로우 설정 화면">

여기서 설정할 수 있는 항목들입니다.

- **Execution Order**: 여러 갈래로 나뉘는 워크플로우의 실행 순서입니다. 특별한 이유가 없으면 기본값(v1)을 유지합니다
- **Error Workflow**: 이 워크플로우가 실패했을 때 대신 실행할 워크플로우입니다. 에러 알림을 보내는 별도 워크플로우를 연결할 수 있습니다
- **This workflow can be called by**: 다른 워크플로우에서 이 워크플로우를 호출할 수 있도록 허용하는 설정입니다
- **Timezone**: 이 워크플로우에서 사용할 타임존입니다. Schedule Trigger의 실행 시각에 직접 영향을 줍니다
- **Save failed production executions**: 실패한 실행 데이터를 저장할지 여부입니다. 기본값은 저장합니다
- **Save successful production executions**: 성공한 실행 데이터를 저장할지 여부입니다. 기본값은 저장합니다
- **Save manual executions**: 에디터에서 수동 실행한 데이터를 저장할지 여부입니다. 기본값은 저장합니다
- **Save execution progress**: 각 노드의 실행 데이터를 저장할지 여부입니다. 저장하면 에러 발생 시 중단된 지점부터 다시 시작할 수 있지만, 실행이 느려질 수 있습니다
- **Timeout Workflow**: 일정 시간이 지나면 실행을 자동으로 취소할지 여부입니다

> [!NOTE]
> **맥락**입니다.

타임존, 에러 워크플로우, 저장 정책.
이것들은 워크플로우의 노드 흐름에는 직접 나타나지 않습니다.
하지만 워크플로우가 어떻게 동작할지를 결정하는 배경 정보입니다.
카페에서 주문서에는 없었지만 처리 방식을 바꿨던 "이 매장의 규칙"과 같습니다.

### Export와 Import — 워크플로우를 내보내고 가져옵니다

n8n 워크플로우는 JSON 파일로 저장할 수 있습니다.
다른 사람에게 공유하거나, 다른 사람이 만든 워크플로우를 가져올 때 사용합니다.

<img src="https://raw.githubusercontent.com/bhcha/new_blog/main/shortform/images/l1-c6-import-export-menu.png" alt="Import/Export 메뉴">

내보내기(Export)와 가져오기(Import)는 세 가지 방법으로 할 수 있습니다.

**에디터 메뉴에서**
- 내보내기: 상단 오른쪽 점 세 개 메뉴에서 Download를 선택합니다. JSON 파일로 다운로드됩니다
- 가져오기: 같은 메뉴에서 Import from URL 또는 Import from File을 선택합니다

**캔버스에서**
- 내보내기: 모든 노드를 선택한 뒤 `Ctrl+C`로 복사합니다. JSON이 클립보드에 들어갑니다
- 가져오기: 복사한 JSON을 캔버스에 `Ctrl+V`로 붙여넣습니다

**명령줄에서**
- n8n CLI 명령어로 워크플로우와 크레덴셜을 내보내거나 가져올 수 있습니다

> 내보낸 JSON 파일에는 크레덴셜 이름과 ID가 포함됩니다. ID 자체는 민감하지 않지만, 크레덴셜 이름은 명명 방식에 따라 민감할 수 있습니다. HTTP Request 노드에 인증 헤더가 포함될 수도 있습니다. 공유하기 전에 이런 정보를 확인하고 제거하거나 익명화하십시오.

자동화는 화면보다 권한과 연결 정보가 더 중요할 때가 많습니다.
그래서 공유 전에 크레덴셜과 민감한 설정을 먼저 확인해야 합니다.

> [!NOTE]
> **입력 → 처리 → 출력** 관점에서 보면, 워크플로우 자체가 하나의 JSON 객체입니다.
> 값의 형태 중 객체는 "이름표가 붙은 값들의 묶음"이었습니다.
워크플로우를 JSON으로 내보내면, 노드 구성, 연결 정보, 설정값이 모두 하나의 객체에 담깁니다.
워크플로우 안에서 흐르는 데이터만 객체가 아니라, 워크플로우 그 자체도 객체입니다.

### 정리하며

Nathan의 워크플로우가 완성되었습니다.

Manual Trigger를 Schedule Trigger로 교체해서 입력 방식을 바꾸었습니다.
Publish로 워크플로우의 상태를 활성으로 전환했습니다.
실행 로그에서 각 실행의 상태를 확인할 수 있게 되었습니다.
Workflow Settings로 타임존과 에러 처리 같은 맥락을 설정했습니다.
Export/Import로 워크플로우를 JSON 객체로 주고받을 수 있게 되었습니다.

이제 Nathan은 매주 월요일 아침, 아무 버튼도 누르지 않아도 됩니다.
워크플로우가 데이터를 가져오고, 필터링하고, 계산하고, 알림을 보내고, 저장합니다.
Nathan은 더 의미 있는 일로 한 주를 시작할 수 있습니다.

### 참고

**JSON(JavaScript Object Notation)**: 데이터를 주고받기 위한 텍스트 형식입니다. 사람이 읽을 수 있고, 거의 모든 프로그래밍 언어와 도구가 이 형식을 지원합니다. n8n의 워크플로우, 노드 간 데이터, 내보내기 파일 모두 JSON 형식입니다.

**크레덴셜(Credential)**: 외부 서비스에 접근하기 위한 인증 정보입니다. API 키, 비밀번호, 토큰 등이 여기에 해당합니다. n8n에서는 크레덴셜을 별도로 관리하며, 워크플로우 내보내기 시 주의가 필요합니다.
