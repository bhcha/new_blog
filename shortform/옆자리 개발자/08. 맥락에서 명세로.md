# 맥락에서 명세로

## SDD(Spec-Driven Development)에 대해 다룹니다.

> **AI 시대, 소프트웨어를 이해하는 법** 시리즈

> 원본: [https://sprawlingdev.substack.com/p/201](https://sprawlingdev.substack.com/p/201)

---

**이전 글 「바이브 코딩, 요청과 맥락 사이」에서 요청 뒤의 맥락을 파악하고 전달하는 것이 컨텍스트 엔지니어링이라 했습니다. 하지만 다루지 못한 영역이 있었습니다. 맥락을 저장하고, 일을 쪼개서 관리하는 것. 이것을 채우면 체계가 됩니다.**

---

### 맥락을 문서로 만들면

> 본 글은 Ryan Carson의 3-File System을 바탕으로 설명합니다. GitHub Spec kit 등 대표적인 여러 방법이 있지만, 비개발자의 시각에서 이해하기 좋은 방법이 3-File System이라 생각해 해당 내용을 바탕으로 설명합니다. 개념 설명만 다루고 실습은 별도로 다뤄 보겠습니다.

이전 글에서 맥락을 파악할 때 어떤 정보를 넣어줄지 골랐고, 가이드라인을 만들었고, 넣지 않아도 될 것을 걸렀습니다.

비개발자의 핵심 요구사항과 개발자 시선에서의 제약 조건, 기술 스펙을 문서로 만들면, 그것을 소프트웨어 개발 문서 중 PRD(Product Requirements Document)라고 합니다.

무엇을 만들지, 어떤 조건에서 만들지, 무엇은 하지 않아도 되는지, 「바이브 코딩, 요청과 맥락 사이」에서 비개발자의 시선이 정의한 “무엇을”과 개발자의 시선이 좁힌 “어떻게”가 PRD에 담깁니다.

PRD는 코딩 에이전트에게 주는 작업 지시서입니다. 명세가 있으면 코딩 에이전트는 추측이 줄어듭니다. 기대값에 가까워지게 만들어질 확률이 높아집니다.

---

### 이전 글에서 다루지 못한 것

이전 글의 부록에서 컨텍스트 엔지니어링의 네 가지 영역을 소개했습니다.

- **선택**: 어떤 정보를 넣고 뺄지 고르는 것.
- **압축**: 긴 정보를 핵심만 남기는 것.
- **저장**: 컨텍스트 윈도우에 다 담지 못하는 정보를 외부에 저장하고, 필요할 때 가져오는 것.
- **분리**: 하나의 큰 작업을 여러 작업으로 나누어 처리하는 것.

PRD를 작성하는 과정은 선택과 압축입니다. 그리고 이것을 파일로 만드는 것 자체가 저장입니다.

하지만 PRD만으로는 부족합니다. 코딩 에이전트는 어디서부터 시작할지, 어디까지 했는지, 무엇이 남았는지 지속적으로 관리하기 어렵습니다. 컨텍스트 윈도우는 무한하지 않기 때문입니다.

---

### 저장과 분리의 핵심은 작업을 추적관리하는 것입니다

분리를 실천하는 가장 대표적인 방법은 PRD를 작업(Task)단위로 분리하는것이고 외부 파일 저장을 통해 추적관리하는 것입니다. 3D 청사 안내도를 한 번에 만드는 것이 아니라, 실행 가능한 단위로 분해합니다.

- 작업 1. 건물 3D 모델 구현
- 작업 2. 층별 부서 데이터 연결
- 작업 3. 층 클릭 시 부서 정보 표시
- 작업 4. 마우스 인터랙션 (회전, 확대)
- 작업 5. 모바일 대응

이것이 **분리**입니다. 하나의 큰 작업을 코딩 에이전트가 처리할 수 있는 크기로 나눕니다.

**작업별로 실행하고, 진행을 추적합니다.** 작업 1을 코딩 에이전트에게 지시합니다. 완료되면 확인하고, 작업 2로 넘어갑니다. 어디까지 했는지, 다음은 무엇인지, 무엇이 남았는지를 파일에 기록합니다. 이것이 **저장**입니다. 진행 상황을 컨텍스트 윈도우 바깥에 저장하고, 필요할 때 가져옵니다. 코딩 에이전트가 이전 작업의 맥락을 잃어도, 기록이 있으면 이어갈 수 있습니다. 새로운 세션에서 작업을 이어받더라도, 기록을 읽으면 어디서부터 시작할지 알 수 있습니다.

---

### 문서가 체계를 만듭니다

이 흐름을 정리하면 세 개의 문서가 됩니다.

- **PRD.** 무엇을 만들지 정의한 명세. 선택과 압축의 결과를 저장합니다.
- **Task List.** PRD를 실행 가능한 단위로 쪼갠 작업 목록. 분리의 결과를 저장합니다.
- **Task Progress.** 작업별 진행 상황 기록. 실행 중 변하는 상태를 저장합니다.

세 문서 모두 컨텍스트 윈도우 바깥에 정보를 저장하고 필요할 때 불러오는 것입니다. 차이는 무엇을 저장하느냐입니다.

```
선택 + 압축 + 저장 → PRD (요구사항)
분리 + 저장       → Task List (작업 분해)
저장             → Task Progress (진행 상태)
```

명세를 먼저 만들고, 작업을 쪼개고, 진행을 추적하는 것. 이 원칙을 Spec-Driven Development(명세 주도 개발)라 합니다.

> Spec-Driven Development는 IBM, Red Hat 등에서 AI 코딩의 다음 패러다임으로 주목하고 있는 방법론입니다. 코드를 먼저 작성하는 것이 아니라, 명세를 먼저 만들고 그 명세가 전체 과정을 이끕니다.

---

### 명세가 전체를 이끕니다

명세의 힘은 코드 생성에서 끝나지 않습니다.

명세가 있으면 설계와 구현이 하나의 흐름으로 이어집니다. 구현 단계에서는 명세를 기반으로 테스트를 먼저 작성하면 그 테스트를 통과하는 코드를 만드는 방법도 있습니다.

> 이러한 개발 방법을 테스트 주도개발(TDD)라고 합니다.

명세 자체가 문서의 역할을 하기 때문에 별도의 문서화 없이도 무엇을 왜 만들었는지 추적할 수 있습니다.

변경이 생기면 명세를 업데이트합니다. 나머지는 명세를 따라 다시 만들어집니다.

바이브 코딩이 구현 한 단계만 다뤘다면, Spec-Driven Development는 명세를 중심으로 설계부터 구현, 테스트까지를 하나로 엮습니다. 이 전체 과정에는 이름이 있습니다.

다음 글 「전체를 본다는 것」에서 이야기합니다.

---

### 정리하며

컨텍스트 엔지니어링은 `요청 뒤의 맥락을 파악하고 전달하는 것`이었습니다.

PRD는 파악한 맥락을 선택하고 압축하여 저장한 것입니다. Task List는 분리를, Task Progress는 실행 상태의 저장을 채웁니다.

Spec-Driven Development는 이 체계 위에서 `명세가 전체를 이끌게 하는 것`입니다.

맥락을 파악하고 전달하는 과정에 체계가 생기면, 결과물은 기대값에 안정적으로 가까워집니다.
