# 데이터의 구조를 이해한다

**n8n에서 노드 사이를 오가는 데이터는 정해진 구조가 있습니다.**
**이 구조를 알면 데이터가 왜 그렇게 보이는지, 왜 아이템 수가 갑자기 바뀌는지 이해할 수 있습니다.**

이번 편에서는 지금까지 여러 번 지나왔던 데이터를 더 정확히 들여다봅니다.
핵심은 `값`, `객체`, `배열`, 그리고 `아이템의 수`입니다.
노드가 왜 아이템마다 동작하는지, 왜 어떤 순간에는 아이템 수가 달라지는지를 구조의 관점에서 설명하겠습니다.

---

### n8n의 데이터는 아이템의 배열이다

n8n에서 노드 사이를 흐르는 모든 데이터는 하나의 형식을 따릅니다.
**배열 안의 객체**입니다.

```json
[
  {
    "json": {
      "name": "홍길동",
      "email": "hong@example.com"
    }
  },
  {
    "json": {
      "name": "김철수",
      "email": "kim@example.com"
    }
  }
]
```

> [!NOTE]
> 카페 주문서로 설명한 것과 같은 구조입니다.

주문서 한 장이 객체였습니다. 대기 중인 주문서 여러 장이 배열이었습니다.
n8n에서는 이 배열 안의 객체 하나하나를 **아이템**이라고 부릅니다.

노드는 들어온 아이템을 하나씩 처리합니다.
아이템이 3개면 3번 처리하고, 결과도 3개가 나갑니다.
**입력 → 처리 → 출력**이 아이템 단위로 반복됩니다.

### json 키가 감싸고 있다

구조를 자세히 보면, 각 아이템이 `json`이라는 키로 한 번 더 감싸져 있습니다.

```json
[
  {
    "json": {
      "apple": "beets",
      "carrot": {
        "dill": 1
      }
    }
  }
]
```

n8n이 데이터를 인식하려면 이 `json` 키가 필요합니다.
Code 노드에서 직접 데이터를 만들 때도 이 형식을 따릅니다.

다만 `json` 키를 빠뜨려도 n8n이 자동으로 추가해 줍니다.
형식을 알아두되, 실수해도 괜찮습니다.

파일이나 이미지 같은 바이너리 데이터는 `json` 대신 `binary` 키를 사용합니다.
텍스트 데이터는 `json`, 파일 데이터는 `binary`라고 이해하면 됩니다.

### 아이템 하나는 객체다

아이템 하나를 들여다보면 객체입니다.
키(key)와 값(value)의 쌍으로 이루어져 있습니다.

```json
{
  "name": "홍길동",
  "email": "hong@example.com"
}
```

"이 아이템의 이름이 뭐지?" — `name`이라는 키로 찾습니다.

> [!NOTE]
> 주문서의 이름표를 키라고 불렀던 것과 같습니다.

n8n에서는 이 키를 **dot notation**으로 참조합니다.
`{{ $json.name }}`이라고 쓰면 현재 아이템의 `name` 값을 가져옵니다.

객체 안에 또 다른 객체가 있을 수도 있습니다.

```json
{
  "name": "홍길동",
  "contact": {
    "personal": "hong@gmail.com",
    "work": "hong@company.com"
  }
}
```

이때 업무용 이메일을 꺼내려면 `{{ $json.contact.work }}`라고 씁니다.
점(`.`)으로 한 단계씩 들어가는 것입니다. 그래서 dot notation이라고 부릅니다.

---

### 노드가 아이템을 처리하는 방식

노드는 아이템마다 같은 작업을 수행합니다.

예를 들어 Trello 노드를 "카드 만들기"로 설정하고, 카드 이름을 `{{ $json.name }}`으로 지정했다고 합시다.

입력이 이렇게 들어오면:

```json
[
  { "json": { "name": "보고서 작성" } },
  { "json": { "name": "회의 준비" } }
]
```

노드는 아이템마다 한 번씩 실행됩니다.
"보고서 작성" 카드 하나, "회의 준비" 카드 하나. 총 두 개의 카드가 만들어집니다.

입력 2개 → 처리 2번 → 출력 2개.
**입력 → 처리 → 출력**이 아이템 단위로 작동한다는 뜻입니다.

---

### 아이템 수가 변할 때

대부분의 노드는 들어온 아이템 수만큼 나갑니다.
그런데 아이템 수가 달라지는 경우가 있습니다.

하나의 아이템 안에 배열이 들어 있을 때입니다.

```json
[
  {
    "json": {
      "team": "마케팅",
      "members": ["김철수", "이영희", "박민수"]
    }
  }
]
```

이 데이터는 아이템 1개입니다.
하지만 `members` 안에는 3명의 이름이 들어 있습니다.

각 멤버에게 개별적으로 메시지를 보내야 한다면, 아이템 1개로는 안 됩니다.
3개의 아이템으로 펼쳐야 합니다.

비개발자가 혼란스러워하는 지점이 여기입니다.
"아이템이 1개였는데 왜 갑자기 3개가 되었지?"

**입력 → 처리 → 출력**으로 보면 명확합니다.
입력은 1개. 처리에서 배열을 펼쳤습니다. 출력은 3개.
처리의 성격이 "펼치기"이기 때문에 아이템 수가 달라진 것입니다.

---

### Split Out 노드 — 하나를 여러 개로 펼친다

**Split Out** 노드는 하나의 아이템 안에 있는 배열을 여러 아이템으로 분리합니다.

위의 예시에서 Split Out 노드를 `members` 필드에 적용하면 이렇게 됩니다.

입력: 아이템 1개 (members 배열 안에 3명)
출력: 아이템 3개 (각각 한 명씩)

이것은 **변환**입니다.
데이터의 내용은 바뀌지 않았습니다. 구조가 바뀌었습니다.
하나의 묶음을 풀어서 개별 아이템으로 만든 것입니다.

코드 없이 노드 하나로 이 변환을 수행할 수 있습니다.

---

### Aggregate 노드 — 여러 개를 하나로 모은다

**Aggregate** 노드는 Split Out의 반대입니다.
여러 아이템을 하나의 아이템으로 합칩니다.

아이템이 3개 있다면, Aggregate 노드를 거치면 1개가 됩니다.
3개의 값이 하나의 배열로 묶입니다.

이것도 **변환**입니다.
개별 아이템들을 하나의 묶음으로 모은 것입니다.

Split Out이 펼치기라면, Aggregate는 모으기입니다.
둘 다 데이터의 구조를 바꾸는 변환입니다.

---

### Code 노드로 데이터를 변환한다

Split Out과 Aggregate 노드로 해결되지 않는 변환은 Code 노드를 사용합니다.

하나의 아이템 안 배열을 여러 아이템으로 펼치는 코드입니다.

```javascript
return $input.first().json.data.map(item => {
    return {
        json: item
    }
});
```

`$input.first()`로 첫 번째 아이템을 가져옵니다.
그 안의 `data` 배열을 하나씩 꺼내서 각각 아이템으로 만듭니다.
Split Out 노드가 하는 것과 같은 일입니다.

여러 아이템을 하나로 모으는 코드입니다.

```javascript
return [
  {
    json: {
      data_object: $input.all().map(item => item.json)
    }
  }
];
```

`$input.all()`로 모든 아이템을 가져옵니다.
하나의 배열로 묶어서 아이템 1개로 만듭니다.
Aggregate 노드가 하는 것과 같은 일입니다.

두 경우 모두 **변환**입니다.
데이터의 내용이 아니라 구조를 바꾸는 것입니다.

---

### Code 노드로 데이터를 직접 만든다

Code 노드를 사용하면 데이터를 직접 정의할 수도 있습니다.

```javascript
return [
  {
    json: {
      name: "홍길동",
      email: {
        personal: "hong@gmail.com",
        work: "hong@company.com"
      }
    }
  },
  {
    json: {
      name: "김철수",
      email: {
        personal: "kim@gmail.com",
        work: "kim@company.com"
      }
    }
  }
];
```

배열 안에 객체를 넣고, 각 객체를 `json` 키로 감쌌습니다.
n8n의 데이터 구조를 그대로 따른 것입니다.

이렇게 만든 데이터는 다음 노드에서 `{{ $json.name }}`이나 `{{ $json.email.work }}`로 참조할 수 있습니다.

테스트용 데이터를 만들거나, 외부에서 가져온 데이터를 n8n 구조에 맞게 정리할 때 유용합니다.

---

### 정리하며

n8n의 데이터 구조는 하나입니다. **아이템의 배열**입니다.

```
[{ json: {...} }, { json: {...} }, { json: {...} }]
```

> [!NOTE]
> 카페 주문서로 설명한 것과 같습니다.

주문서 한 장이 아이템(객체)이고, 대기 중인 주문서 묶음이 배열입니다.

아이템 안의 값은 키로 찾습니다. dot notation으로 꺼냅니다.
노드는 아이템 하나씩 **입력 → 처리 → 출력**을 수행합니다.

아이템 수가 변하는 것은 **변환** 때문입니다.
Split Out은 하나를 여러 개로 펼치고, Aggregate는 여러 개를 하나로 모읍니다.
Code 노드로도 같은 변환을 할 수 있습니다.

데이터의 구조를 알면, 노드 사이에서 무엇이 오가는지 읽을 수 있습니다.
