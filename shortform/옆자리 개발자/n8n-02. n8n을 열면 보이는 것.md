# n8n을 열면 보이는 것

**n8n을 처음 열면 빈 화면이 나옵니다. 이 화면이 무엇인지, 여기에 놓이는 것들이 무엇인지를 설명합니다.**
**공식문서를 바탕으로, 지난 글에서 정리한 핵심 단어가 n8n에서는 어떻게 보이는지도 함께 확인합니다.**

이번 편에서는 지난 글에서 정리한 핵심 단어가 n8n 화면에서 어떻게 보이는지 먼저 잡아봅니다.
캔버스는 `입력 → 처리 → 출력`을 그리는 곳이고, 노드는 그 흐름을 이루는 단계입니다.
특히 `입력`, `질의 / 명령`, `변환 / 선택`이 어떤 노드로 나타나는지 중심으로 보겠습니다.

---

### 시작하기

n8n을 쓰려면 먼저 환경을 준비해야 합니다.

가장 간단한 방법은 **n8n Cloud**입니다. 설치 없이 바로 쓸 수 있고, 무료 체험이 포함되어 있습니다.

> **다른 방법**: n8n Cloud가 맞지 않으면 Docker로 직접 설치할 수도 있습니다. 다만 이 방법은 호스팅, Docker, 명령줄에 익숙한 분에게 더 적합합니다. 설치법은 따로 다루지 않겠습니다.

n8n이 실행되면 브라우저에서 에디터 화면을 엽니다.
로그인 후 **Overview**에서 **Create Workflow**를 선택하면 메인 캔버스가 나타납니다.

여기서 중요한 것은 설치 방식 자체가 아닙니다.
캔버스에서 입력, 처리, 출력의 흐름이 눈에 보인다는 점입니다.
비개발자에게 n8n이 쉬운 이유도 바로 여기에 있습니다.

### 캔버스 — 처리 과정을 눈으로 보는 곳

캔버스는 에디터 화면의 회색 점선 배경입니다.
워크플로우를 만드는 공간입니다.

캔버스에는 몇 가지 요소가 있습니다.

- 화면 확대/축소, 줌 초기화, 노드 정렬 버튼
- **Execute workflow** 버튼 — 첫 노드를 추가하면 나타납니다. 캔버스 위의 모든 노드를 순서대로 실행합니다
- **+** 버튼 — 노드 패널을 엽니다
- **메모** 아이콘 — 캔버스에 스티키 노트를 추가합니다
- **Ask Assistant** 버튼 — AI 어시스턴트에게 도움을 요청할 수 있습니다
- "Add first step" 점선 사각형 — 첫 번째 노드를 추가하는 곳입니다

<img src="https://raw.githubusercontent.com/bhcha/new_blog/main/shortform/images/ch1-canvas.png" alt="캔버스">

> **캔버스 이동 방법**: `Ctrl+왼쪽 클릭`으로 드래그하거나, 마우스 가운데 버튼으로 드래그하거나, 트랙패드에 두 손가락을 올려 밀면 됩니다.

여기서 노드를 하나씩 놓고 연결하면 워크플로우가 만들어집니다.

> [!NOTE]
> **입력 → 처리 → 출력**을 떠올려 보십시오.
> 캔버스에서 노드를 연결하는 것이 바로 그 구조를 눈으로 그리는 것입니다.
> 첫 노드가 입력을 받고, 중간 노드들이 처리하고, 마지막 노드가 출력합니다. 
> 코드에서는 머릿속으로 그려야 했던 흐름이, 여기서는 선으로 보입니다.

다만 캔버스에 보이는 것은 주로 지금 흐르는 데이터입니다.
그중에는 조건 판단에 쓰이거나 다음 행동을 바꾸는 **상태**도 포함됩니다.
왜 이 자동화가 필요한지, 이 규칙이 맞는지 같은 문제는 캔버스 밖의 **맥락**입니다.
이 경계를 알고 보는 것이 중요합니다.

### 에디터 화면 구성

캔버스 외에 두 가지 영역이 더 있습니다.

**왼쪽 패널**

왼쪽에는 워크플로우 관리를 위한 패널이 있습니다. 작은 화살표 아이콘으로 열고 닫을 수 있습니다.

- **Overview** — 접근 가능한 모든 워크플로우, 인증 정보, 실행 기록이 여기에 있습니다. 새 워크플로우도 여기서 만듭니다
- **Personal** — 기본 개인 프로젝트입니다. 별도 프로젝트를 만들지 않으면 워크플로우와 인증 정보가 여기에 저장됩니다
- **Projects** — 워크플로우와 인증 정보를 그룹으로 묶고, 사용자 역할을 지정할 수 있습니다
- **Admin Panel** — n8n Cloud 전용. 사용량, 결제, 버전 설정을 관리합니다
- **Templates** — 미리 만들어진 워크플로우 모음입니다. 자주 쓰는 케이스를 빠르게 시작할 수 있습니다
- **Variables** — 워크플로우 전체에서 공통으로 쓰는 고정 데이터를 저장합니다
- **Insights** — 워크플로우의 분석과 인사이트를 제공합니다
- **Help** — n8n 제품과 커뮤니티 관련 자료입니다

<img src="https://raw.githubusercontent.com/bhcha/new_blog/main/shortform/images/ch1-side-panel.png" alt="왼쪽 패널">

**상단 바**

- **Workflow Name** — 기본 이름은 "My workflow"입니다. 언제든 바꿀 수 있습니다
- **+ Add Tag** — 태그로 워크플로우를 분류할 수 있습니다. 선택 사항입니다
- **Publish** — 워크플로우를 게시합니다. 기본 상태는 미게시입니다
- **Share** — 다른 사용자와 워크플로우를 공유합니다
- **Save** — 현재 워크플로우를 저장합니다
- **History** — 저장 후 이전 버전을 확인할 수 있습니다

<img src="https://raw.githubusercontent.com/bhcha/new_blog/main/shortform/images/ch1-top-bar.png" alt="상단 바">

### 노드 — 워크플로우의 구성 단위

노드는 워크플로우를 이루는 블록입니다.
각 블록이 서로 다른 기능을 담당하고, 연결하면 하나의 자동화가 완성됩니다.

> [!NOTE]
> 노드 하나는 워크플로우의 개별 단계입니다. 데이터를 (a) 불러오거나, (b) 처리하거나, (c) 보내는 역할을 합니다. 
> 노드 하나도 **입력 → 처리 → 출력** 구조입니다. 노드를 클릭하면 INPUT 탭과 OUTPUT 탭이 보입니다. 왼쪽에서 데이터가 들어오고, 노드가 처리하고, 오른쪽으로 데이터가 나갑니다.

<img src="https://raw.githubusercontent.com/bhcha/new_blog/main/shortform/images/Pasted%20image%2020260323213811.png" alt="Pasted image 20260323213811.png">

n8n은 노드를 기능에 따라 네 가지로 분류합니다.

---

### Trigger 노드 — 워크플로우의 입력

**Trigger 노드**는 워크플로우를 시작하고 초기 데이터를 공급합니다.
<img src="https://raw.githubusercontent.com/bhcha/new_blog/main/shortform/images/Pasted%20image%2020260324123324.png" alt="Pasted image 20260324123324.png">

"무엇이 이 흐름을 시작하는가."
스케줄, 웹훅, 이메일 수신, 외부 서비스의 이벤트 등이 트리거가 됩니다.

> [!NOTE]
> **입력**입니다. 카페에서 키오스크에 주문을 누르는 것이 입력이었습니다. 단순히 값을 넣는것을 떠나 어떠한 이벤트들도 입력이 될 수 있습니다. 이벤트로 입력을 일으키는 대표적인 용어가 Trigger 입니다.

Trigger 노드가 그 역할을 합니다. 워크플로우 전체에 데이터를 처음 넣어주는 노드입니다.

---

### App/Action 노드 — 외부 서비스와의 연결

**App 노드**(또는 Action 노드)는 데이터를 추가, 삭제, 편집하거나, 외부 데이터를 요청하거나, 다른 시스템에서 이벤트를 발생시킵니다.
<img src="https://raw.githubusercontent.com/bhcha/new_blog/main/shortform/images/Pasted%20image%2020260324123350.png" alt="Pasted image 20260324123350.png">

Google Sheets, Slack, Gmail, Notion 같은 외부 서비스와 연결되는 노드입니다.

> [!NOTE]
> **질의** 또는 **명령**입니다.
> 데이터를 가져오기만 하면 질의입니다. 구글시트에서 데이터를 읽는 것처럼, 가져오기만 하고 원본은 바뀌지 않습니다.
> 데이터를 바꾸면 명령입니다. Slack에 메시지를 보내거나, 시트에 행을 추가하거나, 주문 상태를 변경하는 것이 명령입니다.


같은 App 노드라도 어떤 동작(Action)을 선택하느냐에 따라 질의가 될 수도 있고 명령이 될 수도 있습니다.

---

### Core 노드 — 내부에서 데이터를 다루는 노드

**Core 노드**는 특정 외부 서비스에 연결되지 않습니다.
대신 논리 처리, 스케줄링, 범용 API 호출 같은 내부 기능을 제공합니다.<img src="https://raw.githubusercontent.com/bhcha/new_blog/main/shortform/images/Pasted%20image%2020260324125512.png" alt="Pasted image 20260324125512.png">
> [!NOTE]
> **변환**과 **선택**이 여기에 해당합니다.

- **If / Switch** — 조건에 따라 흐름이 갈라집니다. 아이스면 얼음을 넣고, 핫이면 넣지 않았던 것처럼. 이것이 **선택**입니다
- **Set / Edit Fields** — 데이터의 형식을 바꾸거나, 필요한 항목만 골라내거나, 새 값을 만듭니다. 이것이 **변환**입니다
- **Code** — 직접 코드를 작성하여 데이터를 처리합니다. 변환의 또 다른 형태입니다
- **Schedule Trigger** — 정해진 시간에 워크플로우를 시작합니다. Core 노드이면서 Trigger 역할을 합니다

Core 노드는 Trigger 역할을 하기도 하고, App 노드처럼 동작하기도 합니다.
외부 서비스가 아닌, 워크플로우 내부의 처리를 담당하는 노드라고 이해하면 됩니다.

### 노드 찾기

노드 패널은 에디터 화면 오른쪽에 있습니다.
세 가지 방법으로 열 수 있습니다.

- 캔버스 오른쪽 상단의 **+** 아이콘 클릭
- 기존 노드 오른쪽의 **+** 아이콘 클릭 — 해당 노드 뒤에 연결할 노드를 추가합니다
- 키보드에서 **N** 키 누르기

<img src="https://raw.githubusercontent.com/bhcha/new_blog/main/shortform/images/ch1-nodes-panel.gif" alt="노드 패널 드릴다운">

노드 패널에서 첫 번째 노드를 추가할 때는 Trigger 노드 카테고리가 보입니다.
Trigger 노드를 추가한 뒤에는 패널이 바뀌어 다음 카테고리가 나타납니다.

- **Actions in an App** — App/Action 노드. 외부 서비스 연동
- **Data transformation** — Core 노드 중 데이터 가공. Code, Filter, Sort, Edit Fields 등
- **Flow** — Core 노드 중 흐름 제어. If, Switch, Merge, Loop 등
- **Core** — Core 노드 중 범용 유틸리티. HTTP Request, Webhook 등
- **Human in the loop** — Core 노드 중 사람 개입. Form, 승인 워크플로우 등
- **Advanced AI** — Cluster 노드. AI Agent, Chain, Vector Store 등

앞서 설명한 4가지 분류 중 **Core 노드가 UI에서 4개 카테고리로 세분화**되어 보입니다.
분류는 같지만 패널에서는 역할별로 나눠 보여주는 것입니다.

특정 노드를 찾고 싶으면 패널 상단의 검색 입력란을 사용합니다.

### 노드 추가하기

캔버스에 노드를 추가하는 방법은 두 가지입니다.

- 노드 패널에서 원하는 노드를 선택합니다. 캔버스에서 선택된 노드에 자동으로 연결됩니다
- 노드 패널에서 캔버스로 드래그 앤 드롭합니다

---

### 노드 위의 버튼들

노드 위에 마우스를 올리면 세 개의 아이콘이 나타납니다.

- **실행** (재생 아이콘) — 이 노드만 실행합니다
- **비활성화/활성화** (전원 아이콘) — 노드를 끄거나 켭니다
- **삭제** (휴지통 아이콘) — 노드를 제거합니다

말줄임표 아이콘을 누르면 추가 옵션이 있는 컨텍스트 메뉴가 열립니다.

> **워크플로우 이동**: 캔버스 위의 모든 노드를 이동하려면 마우스로 전체 선택하거나 `Ctrl+A`를 누른 뒤, 노드 하나를 잡고 드래그합니다.

### 화살표는 순서가 아니라 아이템의 흐름입니다

캔버스에서 노드를 화살표로 연결한다는 것은, 앞 노드가 끝난 뒤 다음 노드가 실행된다는 뜻만은 아닙니다.
앞 노드의 결과 데이터가 그 화살표를 따라 다음 노드로 넘어간다는 뜻이기도 합니다.

n8n에서는 이 데이터의 한 덩어리를 "아이템(item)"이라고 부릅니다.
주문서 한 장, 설문 응답 한 건, 고객 한 명의 정보가 각각 아이템 하나라고 생각하면 됩니다.

중요한 점은, 대부분의 노드가 들어온 아이템을 **하나씩 처리한다**는 것입니다.
아이템이 5개 들어오면, 메시지를 보내는 노드는 보통 5개의 메시지를 보냅니다.
구글 시트에 행을 추가하는 노드라면 5개의 행을 추가합니다.

반대로 어떤 노드는 여러 아이템을 한 번에 다룹니다.
예를 들어 Code 노드는 설정에 따라 모든 아이템을 한 번에 처리할 수도 있습니다.

그래서 워크플로우를 볼 때는 두 가지를 같이 봐야 합니다.

- 이 노드에 **아이템이 몇 개 들어오는가**
- 이 노드는 그 아이템을 **하나씩 처리하는가, 한꺼번에 처리하는가**

이 감각이 있어야 왜 어떤 노드는 한 번만 동작하고, 어떤 노드는 여러 번 결과를 만드는지 읽힙니다.

예를 들어 네이버 뉴스 검색 API 결과를 가져오면 이렇게 보일 수 있습니다.
<img src="https://raw.githubusercontent.com/bhcha/new_blog/main/shortform/images/Pasted%20image%2020260324122531.png" alt="Pasted image 20260324122531.png">

겉으로 보면 기사 10개가 있으니, 아이템도 10개처럼 느껴집니다.
하지만 n8n의 기준으로 먼저 보면, 지금 화살표를 따라 넘어가는 것은 **아이템 1개**입니다.

왜냐하면 가장 바깥쪽이 배열이고, 그 안에 객체가 하나만 들어 있기 때문입니다.
그 객체 안에 `items`라는 이름의 배열이 들어 있고, 기사 10개는 그 안에 들어 있습니다.

즉, 지금 구조는 이렇습니다.
<img src="https://raw.githubusercontent.com/bhcha/new_blog/main/shortform/images/Pasted%20image%2020260324122850.png" alt="Pasted image 20260324122850.png">

- 화살표로 넘어가는 것: 검색 결과 아이템 1개
- 그 아이템 안에 들어 있는 것: 뉴스 기사 목록 10개

이 차이가 중요합니다.

대부분의 노드는 **아이템 단위로** 동작합니다.
그래서 이 상태 그대로 다음 노드에 보내면, 다음 노드는 "뉴스 기사 10개"를 받는 것이 아니라 "기사 목록이 들어 있는 검색 결과 1개"를 받습니다.

만약 기사 10개를 각각 따로 처리하고 싶다면, `items` 배열을 꺼내서 **아이템 10개로 펼치는 작업**이 한 번 더 필요합니다.
<img src="https://raw.githubusercontent.com/bhcha/new_blog/main/shortform/images/Pasted%20image%2020260324123248.png" alt="Pasted image 20260324123248.png">

> [!NOTE]
> 화살표는 단순한 순서 표시가 아닙니다.
> **아이템이 다음 노드로 흐르는 길**입니다.
> n8n을 이해할 때는 "무슨 노드인가"만큼 "아이템이 몇 개 흐르는가"도 중요합니다.

아이템이 실제로 어떤 구조로 생겼는지, 왜 개수가 달라지는지는 뒤에서 별도로 더 자세히 다룹니다.

---

### 정리하며

n8n을 열면 캔버스가 있고, 캔버스 위에 노드를 놓습니다.
노드를 연결하면 워크플로우가 됩니다.

노드는 네 종류입니다.

- **Trigger 노드** — 흐름을 시작합니다. 워크플로우의 **입력**
- **App/Action 노드** — 외부 서비스에 **질의**하거나 **명령**합니다
- **Core 노드** — 내부에서 데이터를 **변환**하고, 조건에 따라 **선택**합니다
- **Cluster 노드** — AI 워크플로우를 위한 노드 그룹

그리고 노드 하나하나도 **입력 → 처리 → 출력** 구조입니다.
클릭하면 INPUT과 OUTPUT이 보입니다.

지난 글에서 카페 이야기로 설명했던 구조가, n8n에서는 눈에 보이는 형태로 나타납니다.
