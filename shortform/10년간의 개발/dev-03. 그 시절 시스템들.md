# 그 시절 시스템들

## 레거시가 만들어진 이유

> 원본: [https://sprawlingdev.substack.com/p/1d4](https://sprawlingdev.substack.com/p/1d4)

---

**지금은 레거시라 불리는 시스템들도, 만들어질 당시에는 가장 합리적인 선택이었습니다. 그 맥락을 이해하지 못하면, 올바른 개선도 불가능합니다.**

---

### 레거시라는 단어에 대하여

“레거시 시스템”이라는 말은 대부분 부정적으로 쓰입니다.  
낡은 것, 바꿔야 할 것, 기술 부채.  
하지만 모든 레거시는 한때 누군가의 최선이었습니다.

운영팀에 와서 20개가 넘는 시스템을 물려받았을 때, 처음에는 저도 “왜 이렇게 만들었지?”라는 생각이 먼저 들었습니다.  
PLSQL로 가득 찬 비즈니스 로직, 단순 전달자에 불과한 백엔드, jQuery와 eval()로 돌아가는 프론트엔드.

하지만 시간을 들여 맥락을 이해하면서 생각이 바뀌었습니다.  
**당시의 제약 조건 안에서, 이것은 가장 합리적인 선택이었습니다.**

---

### 왜 비즈니스 로직이 DB에 있었는가

2010년대 이전, 기업 내부 시스템의 비즈니스 로직은 대부분 Oracle PLSQL(Stored Procedure)로 작성됐습니다.  
지금 기준으로 보면 이상하게 느껴질 수 있습니다.  
하지만 두 가지 현실적인 이유가 있었습니다.

**첫째, 무중단 운영이 필요했습니다.**

당시 기업용 웹 서버는 대부분 WAS(Web Application Server) 기반이었습니다.  
코드를 수정하면 서버를 재시작해야 했습니다.  
재시작 동안 서비스는 중단됩니다.  
하지만 PLSQL은 다릅니다.  
DB 안에서 프로시저를 수정하면, 서버 재시작 없이 즉시 반영됩니다.

24시간 운영되어야 하는 기업 시스템에서, 이것은 결정적인 장점이었습니다.

**둘째, 비용 구조가 달랐습니다.**

솔직한 이야기를 하겠습니다.  
당시 SQL 개발자의 인건비는 백엔드 개발자보다 낮았습니다.  
학습 곡선도 완만했습니다.  
SELECT, INSERT, UPDATE를 할 수 있으면 프로시저를 작성할 수 있었습니다.  
기업 입장에서 같은 비용으로 더 많은 개발 자원을 확보할 수 있었던 것입니다.

이 두 가지 이유가 합쳐져서, “비즈니스 로직은 DB에, 서버는 전달 역할만”이라는 구조가 자연스럽게 자리 잡았습니다.

---

### 백엔드는 거쳐갈 뿐

이런 구조에서 Java 백엔드의 역할은 제한적이었습니다.  
실제로 하는 일은 세 가지였습니다.

1. **PLSQL과 뷰 사이의 데이터 전달**: MyBatis를 통해 파라미터를 변환하고 결과를 전달하는 것이 주 역할이었습니다.
2. **공통 기능 처리**: 인증, 권한 관리, 예외 처리, 로깅 같은 여러 기능에 걸치는 공통 관심사를 담당했습니다.
3. **뷰와의 협업 중재**: 화면에서 필요한 데이터 형태로 가공하여 전달했습니다.

비즈니스 로직이 백엔드에 있지 않으니, 백엔드 개발자가 도메인을 깊이 이해할 필요도 없었습니다.  
이것이 나중에 “시스템은 있는데 아무도 비즈니스를 모른다”는 문제의 씨앗이 됐습니다.

---

### jQuery와 대량 생산의 시대

프론트엔드는 jQuery의 세상이었습니다.  
채용 공고에도 “jQuery 경험자 우대”가 명시되던 시절입니다.

MPA(Multi Page Application) 방식으로, 모든 화면 전환은 페이지 새로고침이었습니다.  
하지만 기업 내부 시스템에서 이것은 문제가 아니었습니다.  
오히려 정형화된 구조 덕분에 대량 생산이 가능했습니다.

화면 레이아웃은 4가지로 표준화되어 있었습니다.

1. Header + Sidebar + Content + Footer (기본 레이아웃)
2. 검색 전용 레이아웃
3. 입력 전용 레이아웃
4. Master-Slave 레이아웃 (목록-상세 구조)

이 4가지 레이아웃에 Template, Boilerplate Code, Utility Function을 조합하면, 주니어 개발자도 빠르게 화면을 만들 수 있었습니다.  
공장처럼 화면을 찍어내는 구조였습니다.

특이한 부분은 동적 처리 방식이었습니다.  
사용자 상호작용이 필요한 경우, JavaScript 코드 자체를 데이터에 저장하고 eval()로 실행하는 방식을 사용했습니다.  
지금 보면 보안적으로 취약한 방식이지만, 당시에는 기업 내부 시스템이라는 폐쇄적 환경에서 합리적인 선택으로 여겨졌습니다.

---

### 맥락을 이해해야 개선할 수 있다

이 시스템들을 “낡았으니 갈아엎자”고 말하기는 쉽습니다.  
하지만 그것이 정답인 경우는 드뭅니다.

레거시 시스템의 진짜 어려움은 코드의 노후화가 아닙니다.  
**비즈니스 로직이 코드 안에 암묵적으로 녹아있다는 것**입니다.  
프로시저 하나에 수천 줄의 로직이 들어있고, 그 로직이 왜 그렇게 작성됐는지 아는 사람은 이미 퇴사했습니다.

결국 레거시를 개선하려면, 먼저 그것이 왜 그렇게 만들어졌는지를 이해해야 합니다.  
맥락 없는 개선은 또 다른 레거시를 만들 뿐입니다.

다음 글에서는 시야를 넓혀, 개발 경력 동안 “개발 외에” 공부한 것들을 이야기합니다.  
함수형 프로그래밍, DDD, 프로젝트 관리, 자동화 — 따로 배웠지만 결국 하나로 연결된 이야기입니다.
